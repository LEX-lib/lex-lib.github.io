## Known Bugs

**Avatar URL fallback never triggers (HIGH)**
- `src/components/CustomNavBar.vue:50-53`. When `auth.user.name` is `undefined`, `encodeURIComponent(undefined)` returns the string `"undefined"`; the `||` short-circuit applies to the entire concatenated URL, not the seed segment. Avatars render with seed `"undefined"` rather than `"guest"`.
- Fix: `const seed = auth.user?.name ?? "guest"; return \`https://api.dicebear.com/9.x/identicon/svg?seed=${encodeURIComponent(seed)}\`;`

**`enrollmentClosed.value` typed `boolean` but assigned `boolean | null` (MEDIUM)**
- `src/components/projects/gift-exchange/GiftExchangeJoin.vue:46`. `drawingStarted || (deadline && now > deadline)` resolves to `null` when `deadline === null` and `drawingStarted === false`. Vue truthy-checks still pass, but TS inference narrows incorrectly and any explicit `=== false` would fail.
- Fix: `enrollmentClosed.value = drawingStarted || (deadline != null && now > deadline);`

**Derangement retry loop is wasteful, not unsafe (MEDIUM)**
- `src/components/projects/gift-exchange/GiftExchange.vue:140-153` and `GiftExchangeManage.vue:152-166`. The `if (!isValid)` guard correctly prevents a bad save, but with exactly 2 participants the only valid permutation is deterministic — retry is wasted work.
- Fix: deterministic derangement (Sattolo's algorithm or cycle-shift).

**LexTrack save loop never refreshes IDs (MEDIUM)**
- `src/views/LexTrackView.vue:127-165`. New entries call `pb.collection(...).create(item)` without capturing the returned record id, so the next save will re-create them.
- Fix: `const created = await pb.collection(...).create(item); item.id = created.id;`

**`removeSupport`/`removeMeeting`/`removeTask` only mutate local state (HIGH)**
- `src/views/LexTrackView.vue:42-91`. Removes from the `ref` array via `lodash-es/remove` but never calls `pb.collection(...).delete(...)`. After save, the entry is gone from UI but persists in PocketBase.
- Fix: when `item.id` is set, `await pb.collection('dsu_*').delete(item.id)` before splicing.

**Editor descriptions saved as raw HTML (MEDIUM)**
- `src/views/LexTrackView.vue:127-162` saves Quill output verbatim via mappers in `src/lib/pocketbase/`. Sanitisation happens only on read in the dead `LexTrackApp.vue:5`. The live `LexTrackView.vue` does not currently render descriptions via `v-html`, so the immediate XSS surface is small — but any future read-side `v-html` will be unsafe.
- Fix: sanitize on write with `DOMPurify.sanitize(...)` before `create`/`update`.

**`alert()` blocking dialogs in dead `LexTrackApp.vue` (LOW)**
- `LexTrackApp.vue:80, 97, 114`. Code is dead; if revived, blocks the main thread.

## Security Considerations

**Plaintext credentials in `local.jsonc` (CRITICAL)**
- `local.jsonc` (project root) contains `VITE_LOGIN_EMAIL` and `VITE_LOGIN_PASSWORD` in plain text alongside HTTPS key/cert paths. Verified NOT in git history (`git log -- local.jsonc` is empty) and gitignored at `.gitignore:48`. However, the file sits in the working tree — accidental rename/copy or removal of the gitignore rule would leak production-style credentials.
- Mitigation today: `.gitignore:48`.
- Recommendations: (1) rotate the credentials in `local.jsonc` since they have been in plaintext on disk; (2) move secrets into `.env.local` (already gitignored); (3) add a pre-commit secret scanner (`gitleaks`/`trufflehog`); (4) document `.env.example` as the canonical onboarding file.

**Env-injected dev-login credentials shipped to client (HIGH)**
- `env.d.ts:7-8` declares `VITE_LOGIN_EMAIL`/`VITE_LOGIN_PASSWORD`. Vite inlines all `VITE_*` variables at build time into the public JS bundle. If `.env.production` (verified to exist, 79 bytes, `Apr 28 20:02`) ever has them set, real credentials would ship to every visitor.
- Recommendations: strip `VITE_LOGIN_*` from `env.d.ts` and remove call sites; rename to non-`VITE_` prefix if a server-side use exists; add a CI grep that fails if `import.meta.env.VITE_LOGIN_PASSWORD` is referenced under `src/`.

**Auth guard is client-side only (HIGH)**
- `src/router/index.ts:70-79` is the only auth gate. Protected views call PocketBase using the same `pb` client whose token lives in `localStorage`. PocketBase's collection rules are the actual source of truth — if `gift_exchange_participants`/`lobbies`/`dsu_*`/`api_requests` have open `list`/`view` rules, data is reachable without auth.
- Recommendation: audit PocketBase API rules; ensure per-record `@request.auth.id != ""` and per-user filters on every collection.

**Manage page exposed to anyone authenticated (HIGH)**
- `src/components/projects/gift-exchange/GiftExchangeManage.vue:9` defines `isSuperUser = computed(() => authStore.isLoggedIn)` — *any* logged-in user is treated as super-user. The route at `src/router/index.ts:46-50` is also unprotected.
- Recommendations: add `meta: { requiresAuth: true }`; replace `isSuperUser` with an explicit role check (e.g. `users.role === 'admin'` or PocketBase `_superusers`); enforce server-side via collection rules.

**Same `isSuperUser` collision in `GiftExchange.vue` (HIGH)**
- `src/components/projects/gift-exchange/GiftExchange.vue:9` — same alias. Any logged-in user could trigger `generatePairs()` which overwrites `drawn_name` for every participant in the global collection. Component is currently unrouted; remove rather than fix.

**PocketBase filter-string interpolation (MEDIUM)**
- 11 occurrences of template-literal `filter:` strings concatenating user input. PocketBase's filter parser has its own quoting rules; an input containing `"` could inject:
  - `src/components/projects/gift-exchange/GiftExchange.vue:97`
  - `src/components/projects/gift-exchange/GiftExchangeJoin.vue:26, 66`
  - `src/components/projects/gift-exchange/GiftExchangeDraw.vue:31, 45`
  - `src/components/projects/gift-exchange/GiftExchangeManage.vue:60, 75, 140, 226`
  - `src/components/projects/api-playground/ApiPlaygroundApp.vue:275` — gated by `^[a-z0-9]{15}$` regex (safe).
  - `src/views/LexTrackView.vue:104` — `dayjs.format('YYYY-MM-DD')` (safe).
- Mitigation today: `.toUpperCase()` and HTML `maxlength` (not sanitisation).
- Fix: use parameterised filters — `pb.collection(...).getFirstListItem("code = {:code}", { filter: { code: userCode.value } })`.

**`v-html` of API response in API Playground (MEDIUM, currently safe)**
- `src/components/projects/api-playground/ApiPlaygroundApp.vue:894` renders `highlightedJson` via `v-html`. DOMPurify is correctly applied at `:155-158` with allowlist `["span"]` + `["class"]` — currently safe. Risk is regression: future commits removing the DOMPurify call or extending `ALLOWED_TAGS` would let arbitrary 3rd-party API responses XSS the user.
- Fix: add a unit test asserting `<script>` injection in JSON values is stripped.

**Random codes are not cryptographically random (LOW)**
- `Math.random()`-based 4-char participant codes and 6–10-char lobby codes are predictable and have no server-side uniqueness check.
- Files: `src/components/projects/gift-exchange/GiftExchange.vue:51-58`, `GiftExchangeJoin.vue:75-82`, `GiftExchangeManage.vue:83-91`.
- Fix: `crypto.getRandomValues()` plus collision check before insert.

## Performance Bottlenecks

**Whole-bundle PrimeVue + Quill + Leaflet in single SPA (MEDIUM)**
- `unplugin-vue-components` + `PrimeVueResolver` (`vite.config.ts:21-25`) registers every PrimeVue component referenced in any template, plus Quill (~200kb) loaded eagerly via `primevue/editor`.
- Improvement: lazy-load `Editor` only on LexTrack route; consider a lighter editor; restrict the resolver to actually-used components.

**Leaflet — already optimised (LOW)**
- `src/components/projects/larga/LargaApp.vue:69-72` uses dynamic `await import("leaflet")` and the `leaflet` rolldown chunk in `vite.config.ts:37`. Good baseline.

**MonitoX `generatePairs` does N parallel updates with no rollback (MEDIUM)**
- `src/components/projects/gift-exchange/GiftExchange.vue:163-170` and `GiftExchangeManage.vue:175-182` open N parallel HTTP `update`s. Any one failure leaves the lobby in a partial state.
- Improvement: use the PocketBase batch API (0.22+) or move the entire derangement into a server-side hook.

**LexTrack save runs sequentially per category (LOW)**
- `src/views/LexTrackView.vue:128-162` `await`s every create/update inside three sequential `for` loops. Use `Promise.all` per category, or batch.

**`fetchStats` and `fetchParticipants` polled together (LOW)**
- `src/components/projects/gift-exchange/GiftExchange.vue:79-80` could derive the count from the participants array instead of double round-tripping.

**`unplugin-vue-components` registers unused demo components (LOW)**
- Configure `Components({ exclude: [...] })` after dead-code removal to keep `components.d.ts` clean.

## Fragile Areas

**Mini-apps share global state via single `pb` client (HIGH)**
- `src/lib/pocketbase/index.ts`, all four mini-apps in `src/components/projects/`. Logging in for API Playground also unlocks LexTrack and the gift-exchange admin views; logout from one logs out everywhere via `pb.authStore.onChange` (`src/stores/auth.ts:11-13`). No per-app sandbox.
- Implication: any change to `auth.ts` or `pb` is cross-cutting — touch only with full regression on all four apps. Currently zero test coverage.

**`unplugin-vue-components` global registration (MEDIUM)**
- `vite.config.ts:21-25`, `components.d.ts`. Any file using `<DataTable>` adds a `DataTable` global; touching any view template can change `components.d.ts` and produce noisy diffs. Removing a component does not remove its declaration until the file regenerates.

**Repo name vestige (LOW)**
- Repo is named `lex-lib.github.io` — a leftover from the prior GitHub Pages hosting. Active deploy is Vercel via GitHub push integration. The `cp dist/index.html dist/404.html` postscript, the `deploy` script, and the `gh-pages` dependency were dropped in commit `498a893`.
- Optional follow-up: rename the GitHub repository (and its remote URL) to drop the `.github.io` suffix to avoid confusion.

**PocketBase types narrowed only at usage sites (MEDIUM)**
- `src/components/projects/gift-exchange/*` use `lobby = ref<any>(null)` and `participants = ref<any[]>([])`. Renaming a field (`drawn_name` -> `assigned_to`) compiles cleanly, then breaks at runtime with no IDE warning.
- Fix: define typed records under `src/types/gift-exchange/` mirroring `src/types/lextrack/`.

**`watch(selectedDate)` does not run on mount in `LexTrackView.vue` (MEDIUM)**
- `src/views/LexTrackView.vue:99-125`. Initial render shows empty arrays; first fetch only fires when the user changes the date.
- Fix: add `{ immediate: true }`.

## Scaling Limits

**Bundle size (MEDIUM)**
- Single SPA hosting four mini-apps: PrimeVue Aura preset + 30+ components + Quill + Leaflet + DOMPurify + Axios + Pinia + Vue Router + Tailwind. Initial JS likely > 500kb gzipped after rolldown chunking.
- Scaling path: per-route entry chunks (rolldown groups already split `leaflet`/`primevue`/`vendor`); move `Editor`/Quill behind dynamic import inside LexTrack only; consider extracting API Playground to its own deployment.

**PocketBase as single auth + data backend (MEDIUM)**
- All four mini-apps share collections. A spike on MonitoX writes (`generatePairs` with 100 participants -> 100 PUTs in parallel) can saturate write rate-limits.
- Scaling path: move pair-generation server-side into a PocketBase JS hook; rate-limit `generatePairs`.

**Vercel Speed Insights fires in dev (LOW)**
- `src/App.vue:4, 11`. `@vercel/speed-insights` is wired correctly for the active Vercel deploy, but `<SpeedInsights />` reports on every render including `npm run dev`, which beacons noisy metrics to the Vercel project.
- Fix: gate behind `import.meta.env.PROD` (e.g. `<SpeedInsights v-if="import.meta.env.PROD" />`).
