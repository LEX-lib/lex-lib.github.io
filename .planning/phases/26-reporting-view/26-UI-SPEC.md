Bars are assigned palette indices in **sort order** (largest spend → index 0 → primary brand color). This guarantees the visually dominant bar always matches the Grand Total color, reinforcing the hero hierarchy.

Define these new tokens in `src/assets/base.css` (planner task — extend `@theme` and `.my-app-dark` blocks):

| 1 | `--color-chart-2` | `#0a6bba` | `#5ea6e6` | Mid blue |

## Component Inventory

### New components in this phase

#### Reports view (file split decision deferred to planner — see 26-CONTEXT.md Open Question 1)

The planner may implement Reports as inline template inside `ExpensesTab.vue` or as an extracted `ExpensesReportsView.vue` sibling. This UI-SPEC is structure-agnostic; the visual + interaction contract below applies wherever the Reports content lives.

#### Sub-tab shell inside `ExpensesTab.vue`

**Purpose:** Two-tab inner navigation (List | Reports) hosted directly under the "Expenses" top-level tab.

**Component:** PrimeVue `<Tabs>` + `<TabList>` + `<Tab>` + `<TabPanels>` + `<TabPanel>` — same component family as `WallecxApp.vue`'s top-level Tabs (see D-26-3).

**Visual differentiation rule (vs top-level Wallecx Tabs):**
The sub-tab Tabs MUST use the **same PrimeVue Tabs component, same Aura styling, same active-tab underline**, but the sub-tab `<TabList>` MUST be visually **secondary-weight** so users perceive nesting depth. Differentiation is achieved by:

1. **Smaller font size on sub-tab labels:** `text-sm` (14px) — matches top-level Tabs which already render at 14px in Aura, so this is parity not a downsize. Differentiation is structural (containment depth), not typographic.
2. **Tighter horizontal padding:** apply `:deep(.p-tabs-tablist .p-tab) { padding: 0.5rem 0.75rem; }` to the sub-tab Tabs only — gives a more compact pill feel than the parent's default `0.75rem 1rem`.
3. **No icon on sub-tab triggers:** top-level tabs have iconify icons (`mdi:cash-multiple`, etc.); sub-tabs render label only. Pure-text labels make the nesting obvious.
4. **Bottom-border thinner:** apply `:deep(.p-tabs-tablist) { border-bottom-width: 1px; }` to the sub-tab Tabs only (Aura default is also 1px — confirm and keep parity). The visual "rule" that there are exactly two levels of tab nesting is enforced by NOT introducing a third style.

**Rationale (documented for ui-checker):** D-26-3 locks PrimeVue Tabs as the sub-tab control. Introducing SelectButton or a "secondary-severity" variant would add visual vocabulary that doesn't exist elsewhere in Wallecx. The compact-padding + no-icon recipe re-uses existing primitives and prevents user confusion: users see two stacked horizontal tab rows where the inner row is clearly subordinate (less padding, no icons).

**Labels:** `"List"` and `"Reports"` — sentence case, no icons. No emoji.

**Active sub-tab persistence:** Out of scope for this phase (see 26-CONTEXT.md Open Question 5). Default active sub-tab = `"list"`.

#### Period selector (inside Reports sub-tab)

**Purpose:** Horizontal 4-tab control: This Month / This Quarter / This Year / Custom.

**Component:** PrimeVue `<Tabs>` with `scrollable` prop enabled (`<Tabs v-model:value="period" scrollable>`).

**Labels (sentence case, no icons):**

| Value | Label |
|-------|-------|
| `this-month` | `"This Month"` |
| `this-quarter` | `"This Quarter"` |
| `this-year` | `"This Year"` |
| `custom` | `"Custom"` |

**Mobile responsive behavior (≤ 360px viewport):**
The period selector MUST use PrimeVue Tabs' built-in `scrollable` prop. Tabs do NOT wrap to a second row, and labels do NOT shorten. Rationale: PrimeVue Tabs with `scrollable` renders prev/next arrow buttons on overflow — a known, accessible interaction. Wrapping to 2 rows breaks the horizontal-tab metaphor; shortening labels ("Mo" / "Qu" / "Yr") sacrifices clarity for ~30px of space.

**Touch target:** every period tab trigger MUST be `min-h-[44px]` — apply via `:deep(.p-tab) { min-height: 44px; }` scoped to the period Tabs container.

**Visual differentiation from sub-tab shell:** the period selector lives **inside** the Reports panel; visual hierarchy (vertical stacking + indentation context) makes the third level of nesting obvious. No additional styling required beyond the same `text-sm` font and `min-h-[44px]` touch target rule.

#### Custom-range DatePicker pair

**Purpose:** When `period === 'custom'`, render two PrimeVue `<DatePicker>` components side-by-side under the period selector.

**Layout:**
- **Desktop (≥ 640px):** `flex items-end gap-2` — From and To rendered side-by-side on a single row, each `w-40` (160px) with a small text label above (`text-xs text-typo-muted` "From" / "To").
- **Mobile (< 640px):** `flex flex-col gap-2` — From stacks above To, each `w-full`. Labels remain above each picker.

**Markup pattern:**
```html
<div v-if="period === 'custom'" class="flex flex-col gap-1 sm:flex-row sm:items-end sm:gap-2">
  <div class="flex flex-col gap-1 flex-1 sm:flex-none sm:w-40">
    <label class="text-xs" style="color: var(--color-typo-muted)">From</label>
    <DatePicker v-model="customFrom" placeholder="From date" dateFormat="dd M yy" class="w-full min-h-[44px]" />
  </div>
  <div class="flex flex-col gap-1 flex-1 sm:flex-none sm:w-40">
    <label class="text-xs" style="color: var(--color-typo-muted)">To</label>
    <DatePicker v-model="customTo" placeholder="To date" dateFormat="dd M yy" class="w-full min-h-[44px]" />
  </div>
</div>
```

**Visibility:** rendered with `v-if="period === 'custom'"` — completely removed from DOM (not hidden) when other periods are active. Matches D-26-5.

**Defaults on first switch to Custom:** `customFrom = dayjs().startOf('month').toDate()`, `customTo = new Date()` (today). Mirrors D-26-5.

**Validation behavior (planner Open Question 4):** when `From > To`, the chart and Grand Total MUST hide and show an inline error message `"From date must be on or before To date."` styled as `text-sm` `color: var(--color-status-error)`, centered, no icon. No silent swap. No Apply button — picks apply instantly (reactive).

#### Grand Total block

**Purpose:** Hero presentation of the period total spend above the chart.

**Layout (centered horizontally, stacked vertically):**

```html
<div class="flex flex-col items-center gap-1 my-6">
  <span class="text-sm" style="color: var(--color-typo-muted)">
    Total spend — {{ periodNameLabel }}
  </span>
  <span class="text-3xl font-bold" style="color: var(--color-brand-primary)">
    {{ formatCurrency(grandTotal) }}
  </span>
</div>
```

**Vertical rhythm:**
- Gap between label and number: `gap-1` (4px) — tight pairing reinforces label↔value relationship.
- Margin above (separating from period selector / Custom DatePickers): `mt-6` (24px).
- Margin below (separating from chart): `mb-6` (24px).
- Result: total block sits in a 24-32-24 vertical band that anchors the eye between selector and chart.

**Alignment:** centered (`items-center`). Rationale: matches the empty-state centering pattern and creates a visual axis of symmetry through the report view.

#### Per-category horizontal bar chart

**Purpose:** Render the per-category breakdown for the selected period.

**Component:**
```html
<Chart type="bar" :data="chartData" :options="chartOptions" class="w-full" />
```

**Container:** wrap `<Chart>` in `<div class="w-full" :style="{ height: chartHeightPx + 'px' }">` where `chartHeightPx = Math.max(220, categories.length * 36)`. Rationale: Chart.js needs a fixed-height parent; horizontal bars per category need ~36px of vertical space (28px bar + 8px gap) to remain comfortably tappable for the tooltip hit area on mobile.

**Chart.js options (planner reference — final shape decided in plan):**

Recommended for v1: **no datalabels plugin** — D-26-8 says "each bar is labeled with its dollar amount" but the X-axis already labels amounts and the tooltip reveals exact values. Defer datalabels to a polish phase if usability testing shows users miss the per-bar amount.

The ROADMAP-level success criterion #2 ("grand total figure updates correctly") is met by the Grand Total block; D-26-8's "absolute amount only on each bar" is interpreted as "no percentage labels — use the X-axis numeric scale (in $) which IS the absolute amount disclosed alongside each bar."

- Minimum bar thickness floor: 20px (Chart.js `minBarLength: 20` for non-zero values — ensures small-spend categories remain visible at long-tail end).

## Interaction Contracts

### Sub-tab switching (List ↔ Reports)
- Default active sub-tab: `"list"` (D-26-9 implies users come to Expenses for the list first; Reports is a destination view).
- Sub-tab switch is **instant** (no transition animation) — matches PrimeVue Tabs default behavior.
- Reports state (selected period, custom-range dates) MUST persist when user toggles back to List and returns. Source: D-26-10 (sessionStorage persistence implies state outlives even a reload, so a sub-tab switch must trivially preserve it).
- The `ExpensesToolbar` (search/sort/category/date-range) is rendered ONLY in the List sub-tab — never in Reports (D-26-1).

### Period selection
- Default: `this-month` (D-26-9).
- Restored from sessionStorage `wallecx:expense-period` on `onMounted` BEFORE first render (mirrors Phase 25 sort restoration pattern — D-26-10).
- Whitelist validation on read: `VALID_PERIODS = ['this-month', 'this-quarter', 'this-year', 'custom']`. Invalid values → fall back to default.
- Period change triggers an **instant** recompute of Grand Total + chart data. No debounce; no loading skeleton between periods (computation is in-memory on already-loaded `expenses` array).
- A **200ms canvas fade-in animation** runs on each period change via Chart.js `animation.duration: 200`. Rationale: confirms to the user that the chart updated; longer durations feel sluggish on rapid tab switches.

### Custom range
- When user switches to `"Custom"` tab: render From + To DatePickers with defaults from D-26-5 (`customFrom = startOf('month')`, `customTo = today`). If sessionStorage has prior `wallecx:expense-period-from` / `wallecx:expense-period-to`, restore those instead.
- Date picks apply **instantly** (no Apply button). Chart + Grand Total recompute on each `@update:model-value`.
- **Invalid range (`From > To`):** chart and Grand Total hide; show inline error `"From date must be on or before To date."` styled `text-sm` `color: var(--color-status-error)`, centered.
- Switching away from `"Custom"` tab and back preserves the From/To selections (state lives in refs).

### Chart hover / tap
- Hover on desktop: bar highlights via `hoverBackgroundColor` (15% lighter/darker depending on theme); tooltip appears near cursor showing `"<Category>: $XX.XX"`.
- Tap on mobile: same tooltip behavior (Chart.js `events: ['touchstart', 'touchmove', 'click']` defaults).
- Tooltip styling: see Color section table (background = `--color-surface-card`, body text = `--color-typo-body`, border = `--color-surface-divider`, 1px border, 4px radius).
- **No drill-down on bar tap** — D-26 deferred ideas list explicitly excludes this. Tapping a bar shows the tooltip; nothing else happens.

### Loading skeleton (Reports sub-tab)
While the parent `ExpensesTab.vue` is in `isLoading === true` (initial `getFullList`):
- The sub-tab shell (List | Reports Tabs) renders normally — users can preview both tabs while data loads.
- Inside the Reports panel during loading:
  - Period selector renders normally (interactive — defaults / restored values apply).
  - Grand Total block shows a `<Skeleton width="12rem" height="2.5rem" />` (label) above a `<Skeleton width="8rem" height="3rem" />` (number), centered.
  - Chart area shows `<Skeleton width="100%" height="220px" class="rounded" />` — single block, no per-bar skeletons.
- On `isLoading === false`: skeletons are replaced by real Grand Total + chart in a 200ms fade (matches Chart.js animation duration for visual coherence).

**Source:** mirrors Phase 25 D-17 skeleton pattern; resolves 26-CONTEXT.md ambiguity ("Block skeleton for chart area, or hide everything behind isLoading?") in favor of block-skeleton-for-chart (users see the period selector immediately).

### Touch targets
Every interactive element in Reports MUST be `min-h-[44px]`:
- Sub-tab triggers (List, Reports) — apply `min-height: 44px` via scoped `:deep()`
- Period tab triggers (Month/Quarter/Year/Custom) — same approach
- DatePicker inputs — `class="min-h-[44px]"`
- Empty-state "Add expense" Button — PrimeVue `size="small"` renders at 32px height; override to `min-h-[44px]` via class
Source: Phase 15 invariant, Phase 25 UI-SPEC.

### Interaction states (hover / focus / active / disabled)
- **Period tab hover (light):** background `color-mix(in srgb, var(--color-surface-page) 92%, var(--color-brand-primary) 8%)` — barely-tinted hover.
- **Period tab hover (dark):** background `color-mix(in srgb, var(--color-surface-card) 92%, var(--color-brand-primary) 8%)`.
- **Period tab focus-visible:** 2px outline `outline: 2px solid var(--color-brand-primary); outline-offset: 2px;` — applies in both themes (brand color shifts automatically).
- **Period tab active (selected):** PrimeVue Aura default — bottom border 2px `var(--color-brand-primary)`, label color `var(--color-brand-primary)`, weight 500.
- **DatePicker focus-visible:** 2px outline `outline: 2px solid var(--color-brand-primary); outline-offset: 0;` — inset matches PrimeVue input convention.
- **Chart bar hover:** see "Chart hover / tap" above (15% lightness shift).
- **No disabled states in this phase.** Apply button is intentionally not introduced; period tabs are never disabled.

### Transition animation on period switch
- Chart canvas: 200ms Chart.js `animation.duration` (configured in chartOptions).
- Grand Total number: **no transition** — the number swaps instantly. Rationale: animating a currency number is distracting and risks transient illegible mid-frames (e.g., "$1,2X4.5X" tweens).
- Period name label: **no transition** — swaps instantly with the number.
- Custom-range DatePicker pair: standard Vue `v-if` mount/unmount, no `<Transition>` wrapper (avoids height-collapse jank on tab switch).

## Dark Mode Contract

Source: `src/assets/base.css` `@theme` light + `.my-app-dark` override blocks; Phase 18 / 22 / 25 patterns.

### Auto-swapping (no JS required)
Every CSS-variable reference in Vue templates and `<style scoped>` blocks MUST auto-swap when `.my-app-dark` is added to `<body>`. The following Phase 26 elements rely on this:

| Element | CSS variable | Light → Dark |
|---------|-------------|--------------|
| Grand Total number | `--color-brand-primary` | `#002244` → `#7896ba` |
| Period name label | `--color-typo-muted` | `#6b7280` → `#9ca3af` |
| Empty-state icon | `--color-brand-primary` | `#002244` → `#7896ba` |
| Empty-state heading | `--color-typo-heading` | `#0d1117` → `#f5f7fa` |
| Empty-state body | `--color-typo-muted` | `#6b7280` → `#9ca3af` |
| DatePicker labels ("From" / "To") | `--color-typo-muted` | `#6b7280` → `#9ca3af` |
| Inline error text | `--color-status-error` | `#c0392b` → `#c0392b` (unchanged) |
| Sub-tab and period tab focus outline | `--color-brand-primary` | `#002244` → `#7896ba` |

### Chart.js dark-mode handling (REQUIRES JS)
Chart.js renders to a `<canvas>`. CSS variables on the parent element DO NOT propagate into canvas drawing — Chart.js needs literal color strings at draw time.

**Required pattern (planner — implement in chartOptions builder):**

1. Define a `getCssVar(name: string): string` helper:
   ```ts
   function getCssVar(name: string): string {
     return getComputedStyle(document.documentElement).getPropertyValue(name).trim()
   }
   ```
2. Build `chartOptions` as a **computed** that reads `getCssVar('--color-chart-1')` etc. at evaluation time.
3. Watch the dark-mode toggle: when `.my-app-dark` class on `<body>` changes, invalidate `chartOptions` so PrimeVue Chart re-renders with new colors.

**Watcher implementation (planner reference):**
```ts
import { useMutationObserver } from '@vueuse/core' // NOT installed per Phase 25 RESEARCH — fall back to manual MutationObserver

const darkModeVersion = ref(0)
onMounted(() => {
  const observer = new MutationObserver(() => { darkModeVersion.value++ })
  observer.observe(document.body, { attributes: true, attributeFilter: ['class'] })
  onBeforeUnmount(() => observer.disconnect())
})

const chartOptions = computed(() => {
  // eslint-disable-next-line @typescript-eslint/no-unused-expressions
  darkModeVersion.value  // make options re-evaluate when class changes
  return { /* options reading getCssVar(...) */ }
})
```

**Alternative (simpler) pattern:** if PrimeVue's `usePrimeVue()` exposes a theme reactivity hook, prefer it. Researcher / planner confirms which approach is simplest in the Phase 26 RESEARCH.md.

### Bar palette under dark mode
The `--color-chart-1` through `--color-chart-8` tokens MUST be added to BOTH the `@theme` block AND the `.my-app-dark` block in `src/assets/base.css` (planner task in Phase 26 plan 01). Without dark-mode entries, the palette defaults to light values and the chart will be unreadable in dark mode.

### Validation
- [ ] Toggling `.my-app-dark` on `<body>` (e.g., via the existing dark-mode switcher) MUST cause the chart to re-render with the dark palette within 100ms.
- [ ] No element in the Reports view MAY appear with mismatched theming (e.g., a light-mode bar on a dark surface) at any point during the toggle.
