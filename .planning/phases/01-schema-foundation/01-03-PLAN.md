  <how-to-verify>
    Step-by-step:

    1. **Open the PocketBase admin dashboard** in a browser at `<VITE_API_BASE_URL>/_/` and log in as a superuser/admin. (If you don't have superuser credentials, you cannot complete this phase — the verify script needs them too.)

    2. **Open the migration doc** in a second window/tab: `.planning/pocketbase-schema.md`.

    3. **Apply Section 1 — Add `duration_unit` to `dsu_meetings`:**
       - Follow the numbered steps verbatim. Field type: Select. Field name: `duration_unit`. Max select: 1. Values: `minutes`, `hours`. Required: unchecked.
       - The doc's "Verify first" preamble lets you skip safely if you've already applied it.

    4. **Apply Section 2 — Add `link` to `dsu_supports`:**
       - Field type: URL. Field name: `link`. Required: unchecked.

    5. **Apply Section 3 — Create `dsu_day_status` collection:**
       - Collection name: `dsu_day_status`. Type: Base.
       - Field `date`: Text, required.
       - Field `status`: Select, max 1, values `sl`, `vl`, `holiday`, required.
       - **Rules tab — set ALL FIVE** to `@request.auth.id != ""`. Leaving any rule blank exposes that operation publicly (T-1-01); the doc highlights this as a critical security step.
       - **Indexes tab** — paste exactly:
         ```sql
         CREATE UNIQUE INDEX idx_dsu_day_status_date ON dsu_day_status (date)
         ```

    6. **Confirm `.env.local`** in the project root contains:
       ```
       VITE_API_BASE_URL=<your PocketBase URL>
       VITE_LOGIN_EMAIL=<superuser email>
       VITE_LOGIN_PASSWORD=<superuser password>
       ```
       (`.env.local` is gitignored — already in this project's gitignore.)

    7. **Run `npm install` if node_modules is missing** — per RESEARCH.md Pitfall 4, the smoke script needs `pocketbase` and `tsx` from node_modules.

    8. **Run the smoke test:**
       ```sh
       npm run verify:schema
       ```

    9. **Read the output:**
       - Exit code 0 with all PASS lines = phase complete; type "approved" below.
       - Any FAIL line = the script names exactly which assertion failed (e.g., "FAIL: dsu_day_status.viewRule = '@request.auth.id != \"\"' (got: '')"). Re-open that collection in the admin UI, fix the field/rule/index, re-run the script. Repeat until exit 0.

    Common failure modes and resolutions:
    - **`Failed to authenticate as superuser` errors** -> The `VITE_LOGIN_EMAIL`/`PASSWORD` you set are for a regular user, not a superuser. Use a `_superusers` (PB 0.23+) or admin (older PB) account.
    - **`FAIL: duration_unit values include minutes and hours (got: [])`** -> You saved the select field with no values; reopen and add `minutes` and `hours` (RESEARCH.md Pitfall 1).
    - **`FAIL: dsu_day_status.<rule> = '@request.auth.id != ""' (got: '')`** -> That rule was left blank in the admin UI. Open the collection -> Rules tab -> paste the rule string into the named field.
    - **`FAIL: dsu_day_status has a UNIQUE index referencing date`** -> The index wasn't added or the SQL didn't include the UNIQUE keyword. Open Indexes tab and re-add with the exact SQL from Section 3 of the doc.
    - **`FAIL: dsu_day_status collection does not exist`** -> Section 3 wasn't applied or the collection name is misspelled. Recheck Section 3.
  </how-to-verify>

| Boundary | Description |
|----------|-------------|
| User -> live PocketBase admin UI | Manual click-through is the only mechanism for schema change in this project (D-01). Any rule left blank silently makes that operation publicly accessible. |
| User local environment -> verify-schema.ts | `.env.local` must contain superuser credentials, sourced from the user's password store. Never committed (gitignored). |

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-1-01 | Elevation of Privilege | dsu_day_status access rules (live state) | mitigate | The verify-schema.ts smoke script (built in Plan 02) explicitly asserts each of the five rule fields equals `@request.auth.id != ""`. The phase cannot pass with a blank rule. This task's "how-to-verify" calls out this exact failure mode. |
| T-1-07 | Tampering | Live schema drift after migration | accept | Once applied, schema can be modified out-of-band by anyone with admin access to the PB dashboard. Out of scope for Phase 1; running `npm run verify:schema` before Phase 2 catches drift introduced between phases. |
| T-1-08 | Information Disclosure | Superuser credentials in `.env.local` | mitigate | `.env.local` is gitignored (existing project convention per CLAUDE.md and `codebase/INTEGRATIONS.md` §Environment Configuration). User instructed never to commit. The verify script reads from process.env via Node's `--env-file` flag — no copy made into source-controlled files. |
