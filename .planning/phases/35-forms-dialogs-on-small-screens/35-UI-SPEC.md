## Spacing Scale

All values are multiples of 4. The existing Tailwind 4 / PrimeVue Aura spacing
scale is already in use. Phase 35 adds no new tokens; references to spacing
below use Tailwind class names or CSS pixel equivalents.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px (`gap-1`) | Icon gaps, inline DragHandle pill spacing |
| sm | 8px (`gap-2`) | Button gap within action bar row; action bar bottom padding |
| md | 16px (`space-y-4`) | Between form field rows (existing pattern); action bar top padding |
| xl | 32px | — |
| 2xl | 48px | — |
| 3xl | 64px | — |

Exceptions:
- **44px minimum touch target** on all buttons, labels wrapping file inputs, and
  the Drawer/Dialog close icon — enforced via `.wallecx-root .p-button.p-button-icon-only`
  rule (Phase 34) and the `min-h-[44px]` Tailwind class on raw `<label>` buttons.
- **Action bar bottom padding**: `0.5rem` (8px) flat — NOT `max(env(safe-area-inset-bottom), ...)`.
  The outer `.p-drawer-content` padding-bottom already absorbs the iOS home-indicator
  inset via `max(env(safe-area-inset-bottom), 1.25rem)` (Phase 34). Double-stacking
  the env() would produce ~68px total on a notched iPhone. See Critical Note below.

## Typography

Phase 35 introduces no new type roles. The existing project scale applies throughout:

| Role | Size | Weight | Line Height | Usage in Phase 35 |
|------|------|--------|-------------|-------------------|
| Dialog title / header | 16px (1rem) | 600 (semibold) | 1.2 | BaseMobileDialog `#header` title `<span>` |
| Field label | 14px (0.875rem = `text-sm`) | 400 (regular) | 1.4 | All `<label>` elements inside form fields |
| Body / input text | **16px on mobile (`≤640px`)** / 14px on desktop | 400 | 1.5 | `<input>`, `<textarea>`, Select label — enforced by FD-01 rule |
| Helper / muted | 12px (`text-xs`) | 400 | 1.4 | Char counter, "(optional)" tag, file-selected name, "Current receipt — select a new file to replace it" |

**NFR-IOS-NO-ZOOM enforcement (FD-01):** All form inputs MUST render at ≥16px
on `≤640px` viewports. This is the hard floor — no `text-xs` or `text-sm` class
may appear on any `<input>` element or its PrimeVue wrapper. Audit grep for
`text-xs.*input\|text-sm.*input` must return 0.

Sources: `ManageExpense.vue` label/text-sm pattern confirmed; `base.css` type scale confirmed.

## Contract 1 — BaseMobileDialog Layout

**Source decisions:** D-35-01, D-35-02, D-35-03 (locked)

### Slot Contract

```
BaseMobileDialog props:
  v-model:visible  Boolean (required)
  title            String (required)
  :is-dirty        Boolean (required — drives dirty guard)
  :is-saving       Boolean (required — disables close icon + close actions)

Slots:
  #default         Form body — rendered ONCE in child, never duplicated
  #actions         Save/Cancel buttons — routed to sticky bar (mobile)
                   or Dialog #footer (desktop)

Exposed method:
  closeWithoutGuard()   Called by child Save/Cancel handlers to bypass dirty guard
```

### Mobile Shell (Drawer, `isMobile === true`, `≤639px`)

| Property | Value | Source |
|----------|-------|--------|
| Component | `<Drawer position="bottom">` | D-35-01 |
| Height | `85dvh` (enforced by `.p-drawer-bottom .p-drawer` in `wallecx-overrides.css`) | Phase 34 substrate |
| `:dismissable` | `false` — prevents auto-close; all close paths go through `@before-hide` | D-35-09 |
| `:show-close-icon` | `!isSaving` — hidden while save is in flight | ManageExpense pattern |
| `@before-hide` | `onBeforeHide()` — dirty guard intercept | D-35-09 |
| `@hide` | `onHide()` — resets `_bypassGuard` flag | D-35-09 |

**Header slot (`#header`):**
- Flex column, center-aligned, full width, `gap-1` (4px)
- `<DragHandle />` — `w-8 h-1 rounded-full bg-gray-300 dark:bg-gray-600` decorative pill (Phase 34)
- `<span class="font-semibold">{{ title }}</span>` — 16px/600 title

**Body (`#default` slot rendering):**
- Container: `<div ref="formRef" @focusin="onFocusin">` — focusin auto-scroll listener (FD-06)
- `class="space-y-4"` (16px between field rows) — mirrors existing form pattern
- Content scrolls inside `.p-drawer-content` (PrimeVue's `overflow-y: auto` scroll container)
- Safe-area padding already applied to `.p-drawer-content` by Phase 34 rule

**Actions slot rendering (mobile):**
- Wrapped in `<div class="wallecx-manage-actions">` directly after `<slot />` — inside the scrollable `<div ref="formRef">` wrapper, NOT inside a PrimeVue footer slot
- Sticky via CSS class `.wallecx-manage-actions` (see Contract 2 below)

### Desktop Shell (Dialog, `isMobile === false`, `≥640px`)

| Property | Value | Source |
|----------|-------|--------|
| Component | `<Dialog modal>` | D-35-01 |
| Width | `40vw` with breakpoints `{ '960px': '75vw', '641px': '92vw' }` | ManageExpense existing pattern |
| `:closable` | `!isSaving` | ManageExpense pattern |
| Content height | `max-height: 80dvh; overflow-y: auto` — enforced by `.p-dialog-content` in `wallecx-overrides.css` | Phase MOB-05 substrate |

**Actions slot routing (desktop):** Rendered in Dialog's `<template #footer>` slot — outside the scroll area, always visible at the bottom of the dialog.

**Dark mode:** Both shells inherit dark surface via `ManageExpense.vue` scoped deep overrides pattern:
```css
:deep(.my-app-dark .p-dialog),
:deep(.my-app-dark .p-dialog .p-dialog-content) {
  background-color: var(--color-surface-card);
  color: var(--color-typo-body);
}
:deep(.my-app-dark .p-drawer),
:deep(.my-app-dark .p-drawer .p-drawer-content) {
  background-color: var(--color-surface-card);
  color: var(--color-typo-body);
}
```
These scoped `:deep` rules should live in `BaseMobileDialog.vue`'s `<style scoped>` block after extraction.

## Contract 2 — Sticky Save/Cancel Action Bar (LT-08)

**Source decisions:** D-35-05, D-35-06 (locked); Pitfall 6 (no safe-area double-stack)

### CSS Rule (add to `wallecx-overrides.css`)

```css
/* Phase 35 LT-08: Sticky Save/Cancel action bar.
 * Sits at the bottom of .p-drawer-content / .p-dialog-content scroll container.
 * Background matches the dialog surface to mask scrolling content.
 *
 * CRITICAL: padding-bottom is a FLAT 0.5rem (8px) — NOT env(safe-area-inset-bottom).
 * The outer .p-drawer-content already pads the bottom via
 * max(env(safe-area-inset-bottom), 1.25rem) from Phase 34. Adding the env() a
 * second time would double the home-indicator clearance (~34px + ~34px = 68px
 * total on a notched iPhone). The existing content padding is sufficient. */
.wallecx-manage-actions {
  position: sticky;
  bottom: 0;
  padding-top: 1rem;
  padding-bottom: 0.5rem;
  background: var(--p-drawer-background, var(--p-overlay-modal-background));
  border-top: 1px solid var(--color-surface-divider);
  z-index: 1;
}

.my-app-dark .wallecx-manage-actions {
  background: var(--color-surface-card);
}
```

### Button Layout (inside `#actions` slot content in child component)

```html
<div class="flex gap-2">
  <Button
    type="button"
    label="Cancel"
    severity="secondary"
    fluid
    :disabled="isSaving"
    @click="onCancel"
  />
  <Button
    type="submit"
    :form="formId"
    :label="isEditMode ? 'Save Changes' : 'Add [noun]'"
    fluid
    :loading="isSaving"
    :disabled="isSaving"
  />
</div>
```

| Property | Value | Rationale |
|----------|-------|-----------|
| Layout | `flex gap-2` (8px between buttons) | Consistent with existing button-group patterns |
| Width | `fluid` on BOTH buttons — each takes 50% of the bar | Full-width on mobile; provides equal-weight affordance |
| Height | Minimum 44px (PrimeVue default Button height at normal size + padding) | NFR touch-target floor |
| Order | Cancel (left/secondary) → Save (right/primary) | Industry-standard OK/Cancel order on mobile |
| Cancel severity | `secondary` — outlined/muted appearance | Distinguishes from primary action |
| Save severity | default (`p-button-primary`) — Aura indigo fill | Primary action emphasis |
| Disabled state | Both disabled during `isSaving`; Save shows loading spinner | Prevents double-submit |

**Desktop (Dialog `#footer`):** Same button markup; `fluid` still works in the footer context.
No sticky CSS needed — Dialog footer is always visible outside the scroll area.

## Contract 3 — FD-01 iOS No-Zoom Rule (16px minimum)

**Source decision:** D-35-12 (locked); NFR-IOS-NO-ZOOM

### CSS Rule (add to `wallecx-overrides.css`)

```css
/* Phase 35 FD-01: iOS auto-zoom prevention.
 * iOS Safari zooms the page when a focused input has font-size < 16px.
 * Overrides PrimeVue Aura's default ~14px (0.875rem) via !important.
 * Non-scoped: this file loads globally for the Wallecx chunk and must reach
 * teleported overlay DOM (DatePicker panel, Select dropdown) outside .wallecx-root.
 * .wallecx-root scoping intentionally NOT applied — teleported elements render
 * outside the root and would be missed. 16px on inputs is benign site-wide. */
@media (max-width: 640px) {
  .p-inputtext,
  .p-inputnumber-input,
  .p-textarea,
  .p-select-label,
  .p-multiselect-label,
  .p-datepicker-input {
    font-size: 16px !important;
  }
}
```

**Selector rationale (all verified against `node_modules/primevue` source):**
- `.p-inputtext` — InputText, editable Select text part, InputNumber's visible input
- `.p-inputnumber-input` — InputNumber's internal `<input>` element
- `.p-textarea` — Textarea's `<textarea>` element
- `.p-select-label` — Select display label (the visible text)
- `.p-multiselect-label` — MultiSelect display label
- `.p-datepicker-input` — DatePicker's InputText element

**Audit gate:** `grep -r "text-xs\|text-sm" src/components/projects/wallecx/ | grep -i "input"` must return 0.

## Contract 4 — FD-04 Inline DatePicker on Mobile (`:inline="isMobile"`)

**Source decision:** D-35-13 CORRECTED (locked) — `touchUI` prop does not exist in PrimeVue 4.5.5. Locked approach: `:inline="isMobile"`.

### Inline Calendar Sizing and Placement

The DatePicker switches from a floating overlay (desktop) to an always-visible
inline grid (mobile) via `:inline="isMobile"`. The inline calendar renders inside
the form's scroll flow, not in a teleported overlay.

| Property | Value | Rationale |
|----------|-------|-----------|
| Binding | `:inline="isMobile"` | Only renders inline on `≤639px`; popup on `≥640px` |
| Width | `fluid` (100% of parent form column) | Fills the form field width |
| Date format | `dateFormat="dd M yy"` (e.g. "27 May 26") | Consistent with existing DatePicker usage |
| `showButtonBar` | `true` on inline instances | Provides "Today" and "Clear" tap targets; recommended for touch |
| Vertical cost | Absorbed by dialog scroll + sticky action bar (D-35-05) | The 85dvh Drawer + `position:sticky` action bar accommodates the grid height (~300px typical) |
| Dark mode | PrimeVue Aura handles dark via `my-app-dark` tokens; no additional CSS needed | Verified via existing DatePicker dark-mode behavior |

**Five DatePicker sites receiving the change (FD-04):**

| File | Field | Notes |
|------|-------|-------|
| `ManageExpense.vue` | Expense date | Direct v-model `expenseDate` |
| `ExpensesToolbar.vue` | From date filter | Uses `@update:model-value` cast pattern |
| `ExpensesToolbar.vue` | To date filter | Uses `@update:model-value` cast pattern |
| `ExpensesReportsView.vue` | Custom From | v-model `customFrom` |
| `ExpensesReportsView.vue` | Custom To | v-model `customTo` |
| `ManageMembership.vue` | Expiry date | Direct v-model `expiryDate`; collapses into BaseMobileDialog `#default` slot |
| `ManageVaccination.vue` | Date administered | Direct v-model `administeredDate` (D-33-01-A invariant — NOT Form-bound) |

**Inline calendar field group spacing:** Wrap each DatePicker site in:
```html
<div class="flex flex-col gap-1">
  <label class="text-sm" style="color: var(--color-typo-heading)">Date *</label>
  <DatePicker v-model="expenseDate" :inline="isMobile" fluid dateFormat="dd M yy" showButtonBar :disabled="isSaving" />
  <Message v-if="expenseDateError" severity="error" size="small" variant="simple">
    {{ expenseDateError }}
  </Message>
</div>
```
Gap between the label and the inline calendar: `gap-1` (4px) — same as all other form fields.

## Contract 6 — FD-09 Dirty-Guard ConfirmDialog

**Source decisions:** D-35-07, D-35-08, D-35-09 (locked); CON-CONFIRMDIALOG-SINGLETON

### Exact Copy

| Element | Text |
|---------|------|
| Header | `Discard changes?` |
| Message body | `Your unsaved changes will be lost.` |
| Accept label | `Discard` |
| Reject label | `Keep editing` |
| Accept button class | `p-button-danger` (destructive — red fill via Aura) |

### Behavior

| Trigger | Guard fires? | Action |
|---------|-------------|--------|
| Backdrop tap (mobile Drawer, `isMobile === true`) | YES, if `isDirty === true` | Show confirm → Discard closes, Keep editing keeps open |
| Swipe-down gesture (mobile Drawer) | YES, if `isDirty === true` | Same as backdrop tap (both go through `@before-hide`) |
| Esc key (mobile Drawer) | YES, if `isDirty === true` | Same as above |
| Backdrop tap (mobile Drawer), `isDirty === false` | NO | Drawer closes immediately (programmatic `visible.value = false`) |
| Explicit Cancel button click | NO | Calls `closeWithoutGuard()`; Drawer closes immediately |
| Save (onSubmit success path) | NO | Calls `closeWithoutGuard()`; Drawer closes immediately |
| Desktop Dialog close icon | NO | Desktop Dialog retains free-dismiss (D-35-08) |
| Desktop Dialog backdrop | NO | Desktop Dialog retains free-dismiss |

### Singleton Guard

Reuses the single `<ConfirmDialog />` at `WallecxApp.vue` line 105.
BaseMobileDialog calls `useConfirm().require(...)`. NO second `<ConfirmDialog>` instance.
Grep audit `<ConfirmDialog` must return exactly 1 line.

### Dirty Comparison Pattern

`isDirty` is a `computed<boolean>` prop in the CHILD component (not in BaseMobileDialog).
Snapshot taken on `visible` rising edge (`watch(visible, (isOpen) => { if (isOpen) snapshot.value = captureSnapshot() })`).
Comparison: field-by-field equality, or `JSON.stringify` for array data (ManageBudget `localRows`).

## Contract 8 — Input States

Applies to all 4 Manage dialogs in both mobile (Drawer) and desktop (Dialog) contexts.
PrimeVue Aura indigo handles state tokens; the contracts below specify the semantics.

### Default

| Mode | Border | Background | Text |
|------|--------|------------|------|
| Light | `--p-inputtext-border-color` (Aura gray/200) | `--p-inputtext-background` (white) | `--p-inputtext-color` (body dark) |
| Dark | `--p-inputtext-border-color` (Aura gray/700 dark) | `--p-inputtext-background` (surface-dark) | `--p-inputtext-color` (body light) |

### Hover

PrimeVue Aura applies `--p-inputtext-hover-border-color` (slightly darker border) on `:hover`.
No explicit override needed.

### Focus-visible

PrimeVue Aura applies `--p-inputtext-focus-border-color` (indigo) and a `box-shadow` focus ring
(inset 0 0 0 1px `--p-primary-color`) on `:focus-visible`. No explicit override needed.
Executor must NOT suppress PrimeVue's default focus ring — it is the only focus
indicator and removing it is an accessibility regression.

### Error State

Triggered by `<Message severity="error" size="small" variant="simple">`:
- Message renders in `--color-status-error` (#c0392b / same in both themes) below the field
- Input border does NOT change to red (PrimeVue's `invalid` prop on InputText/InputNumber
  may be used optionally, but the existing ManageExpense pattern uses only the
  `<Message>` component — maintain consistency)

### Disabled-During-Save

All inputs and buttons receive `:disabled="isSaving"`.
PrimeVue Aura applies `opacity: 0.6` + `pointer-events: none` on disabled state.
No additional CSS override needed.

### Keyboard-Open Layout (iOS + Android)

**Android:** `interactive-widget=resizes-content` (already in `index.html` Phase 34) shrinks
the viewport. The sticky action bar naturally rises above the keyboard with no JS.

**iOS:** Keyboard overlays the viewport without shrinking. `scrollIntoView({ block: 'center',
behavior: 'smooth' })` called via `requestAnimationFrame` on `focusin` inside BaseMobileDialog's
form container moves the focused input into the visible area. The sticky bar may be partially
under the keyboard on long forms — acceptable for Phase 35; VisualViewport escalation is the
documented fallback in Phase 38 UAT.
