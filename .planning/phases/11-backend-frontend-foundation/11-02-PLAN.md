<objective>
Confirm that per-user isolation holds across all five access types for wallecx_memberships before any frontend work begins.

Purpose: Per D-01, the two-user smoke test is a blocking human-action checkpoint. Output: Developer's written confirmation that User B receives 403 or 404 for all five access attempts against User A's membership card records.
</objective>

    Perform the following steps using two real PocketBase user accounts. The existing API Playground at `/projects/api-playground` can run tests 1–4 directly without writing any code.

    **Setup: Create User A's test record**

    1. Log in to the Lexarium app (`npm run dev` → `/login`) as your regular developer account (User A).
    2. Open the PocketBase Admin UI. Navigate to `wallecx_memberships`. Click "New record".
    3. Fill in:
       - user: (select your User A account)
       - card_name: `Smoke Test Card`
       - issuer: `Test Gym` (optional — any value)
       - barcode_type: `qr` (select from the dropdown — must be one of the 5 SELECT values)
       - Optionally attach any image file (PNG or JPG) to the `card_image` field.
    4. Save the record. Note:
       - The record's `id` (e.g. `abc123xyz`)
       - The `card_image` file URL, which appears in the Admin UI as something like:
         `http://127.0.0.1:8090/api/files/wallecx_memberships/abc123xyz/filename.jpg`
         (If you did not attach an image, skip Test 5 and note that in your report.)

    **Setup: Prepare User B**

    5. If User B does not exist from Phase 1 smoke testing: in PocketBase Admin UI, go to "Users" (or `_pb_users_auth_` collection). Create a second user account — use any email/password (e.g. `userb@test.local` / `testpassword123`).
    6. Log in to the Lexarium app as User B (open an incognito/private window, navigate to `/login`, sign in with User B's credentials).
    7. Obtain User B's auth token: the API Playground at `/projects/api-playground` returns the token in the login response. Set the Authorization header to `Bearer <userB_token>` for tests 1–4.

    **Test 1: List isolation**

    Request: `GET http://127.0.0.1:8090/api/collections/wallecx_memberships/records`
    Auth: User B's token in Authorization header.
    Expected: HTTP 200 with `items: []` (empty array — User B sees zero membership cards).
    FAIL if: `items` contains any records belonging to User A.

    **Test 2: View isolation**

    Request: `GET http://127.0.0.1:8090/api/collections/wallecx_memberships/records/{userA_record_id}`
    Auth: User B's token.
    Expected: HTTP 404 (record not found — collection rule treats it as non-existent for User B).
    FAIL if: HTTP 200 is returned with User A's record data.

    **Test 3: Update isolation**

    Request: `PATCH http://127.0.0.1:8090/api/collections/wallecx_memberships/records/{userA_record_id}`
    Body: `{ "card_name": "HACKED" }`
    Auth: User B's token.
    Expected: HTTP 403 or 404.
    FAIL if: HTTP 200 is returned and the record is modified.

    **Test 4: Delete isolation**

    Request: `DELETE http://127.0.0.1:8090/api/collections/wallecx_memberships/records/{userA_record_id}`
    Auth: User B's token.
    Expected: HTTP 403 or 404.
    FAIL if: HTTP 200 or 204 is returned and the record is deleted.

    **Test 5: card_image file URL isolation**

    Action: Copy the card_image file URL from Step 4 above (e.g. `http://127.0.0.1:8090/api/files/wallecx_memberships/abc123xyz/filename.jpg`). Open this URL in a fresh incognito/private browser tab where you are NOT logged in (no PocketBase session cookie, no `?token=` query parameter).
    Expected: HTTP 403 response (because `protected: true` on the card_image field requires a short-lived token).
    FAIL if: The image downloads or displays successfully without a token.

    NOTE: If you did not attach a card_image in Step 3, note "Test 5: skipped — no image attached" in your resume signal. Attach an image and re-run this test before proceeding to Plan 03 if possible.

    **If any test fails:**

    - Recheck the collection rules in Task 2 of Plan 01 — copy the exact rule strings and save again.
    - Verify the `card_image` field has `protected: true` enabled in PocketBase Admin UI → wallecx_memberships → card_image field settings.
    - Run the failing test again after each fix.
    - Do NOT proceed to Plan 03 until all five tests pass.

    **Cleanup (optional but recommended):**

    After all five tests pass, delete the smoke test record from the Admin UI (User A's `Smoke Test Card` record). User B's account can remain in PocketBase for future testing.
