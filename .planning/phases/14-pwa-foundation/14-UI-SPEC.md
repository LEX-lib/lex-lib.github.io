# Phase 14 — UI Design Contract

> Visual and interaction contract for Phase 14: PWA Foundation.
>
> This is an infrastructure phase. The design system (PrimeVue 4 + Tailwind CSS 4,
> Rubik font, navy/amber theme) is inherited **unchanged** from Phase 3 and carried
> through all subsequent phases. No new visual components, screens, dialogs, or
> layouts are introduced. The entire UI surface of this phase consists of exactly
> two toast notifications fired from `WallecxApp.vue`.
>
> For all base design tokens, spacing, typography, color, and component patterns
> not specified here, use `03-UI-SPEC.md` as the authoritative source.

## Component Specifications

### Phase 14 UI Surface: Two Toast Notifications

Both toasts are triggered from `WallecxApp.vue`. No new `.vue` component files are created.

---

### Toast 1: SW Update Prompt (PWA-06)

**Trigger:** `watch(needRefresh, (val) => { if (!val) return; toast.info(...) })`
where `needRefresh` is the reactive Ref returned by `useRegisterSW()` from `virtual:pwa-register/vue`.

**When it fires:** When a new service worker has finished installing and is waiting to activate. This happens on the second load after a new production deployment. It does NOT fire on first install.

**vue-sonner call:**
```typescript
toast.info("A new version of Wallecx is available.", {
  duration: Infinity,
  action: {
    label: "Refresh",
    onClick: () => updateServiceWorker(true),
  },
  cancel: {
    label: "Later",
    onClick: () => {
      needRefresh.value = false;
    },
  },
})
```

**Interaction contract:**

| Element | Spec |
|---------|------|
| Message | "A new version of Wallecx is available." |
| Action button label | "Refresh" |
| Action button behavior | Calls `updateServiceWorker(true)` — activates the waiting SW and reloads the page |
| Cancel button label | "Later" |
| Cancel button behavior | Sets `needRefresh.value = false` — dismisses the toast; the SW remains waiting; it will prompt again on the next app open |
| Duration | `Infinity` — toast stays until the user explicitly acts; it is never auto-dismissed |
| Severity | `info` — blue, non-blocking |
| Position | Top-right (existing `<Toaster />` position in `App.vue`) |
| Blocking | Non-blocking — CRUD dialogs and all other interactions remain fully usable while the toast is visible |

**Why duration: Infinity:** If the toast auto-dismissed, users who dismissed it inadvertently would lose the update prompt indefinitely (until the next full page reload). A persistent toast ensures they can act when convenient. Source: 14-RESEARCH.md Pattern 3.

**Unsaved data risk:** `registerType: 'prompt'` (LOCKED) guarantees no forced reload. Users clicking "Refresh" while a CRUD dialog is open will lose unsaved data — this is intentional and documented. The "Later" option exists precisely for this scenario. Source: STATE.md locked decisions / 14-RESEARCH.md Pattern 3.

---

### Toast 2: Session Expired (PWA-05)

**Trigger:** `onMounted` lifecycle hook in `WallecxApp.vue`, after `navigator.storage.persist()` completes, when `pb.authStore.isValid` returns `false`.

**When it fires:** When the user opens the installed PWA (or navigates to `/projects/wallecx`) and their PocketBase JWT token has expired — most commonly after the iOS 7-day localStorage eviction window or simply after the standard token lifetime.

**vue-sonner call:**
```typescript
toast.info("Your session has expired. Please sign in again.")
await router.push({ name: "login", query: { redirect: "/projects/wallecx" } })
```

**Interaction contract:**

| Element | Spec |
|---------|------|
| Message | "Your session has expired. Please sign in again." |
| Severity | `info` — blue, informational (not `error` — session expiry is expected behavior, not a failure) |
| Duration | Default (4000ms auto-dismiss) — standard toast duration; the router redirect provides the action path |
| Action button | None — the router redirect to `/login?redirect=/projects/wallecx` is the implicit action |
| Cancel button | None |
| Position | Top-right (existing `<Toaster />` position) |

**Sequence note:** The toast fires before the router redirect. The user sees the toast briefly as they are navigated to the login screen. On login success, the `redirect` query parameter returns them to `/projects/wallecx`.

## Interaction & Motion

### Phase 14 Interaction Summary

This phase introduces no new interaction patterns beyond the two toasts documented above. No new click handlers, no new forms, no new dialogs, no new state machines. The only new user-facing interactions are:

| Interaction | Component | Behavior |
|-------------|-----------|----------|
| Click "Refresh" on SW update toast | `WallecxApp.vue` watch | `updateServiceWorker(true)` → page reloads with new SW active |
| Click "Later" on SW update toast | `WallecxApp.vue` watch | `needRefresh.value = false` → toast dismissed; SW stays waiting |
| See session-expired toast | `WallecxApp.vue` onMounted | Followed immediately by router redirect to `/login?redirect=/projects/wallecx` |

### PWA Install Prompt

The browser's native "Add to Home Screen" prompt (Android Chrome) is triggered automatically when the PWA installability criteria are met — no custom UI is added in Phase 14. Phase 15 adds `PwaInstallBanner.vue` for iOS Safari.

### App Shell Offline Behavior

When the installed PWA is opened without network connectivity, the Workbox-generated SW serves the precached `index.html` (via `navigateFallback: 'index.html'`). The user sees the app shell (Vue Router navigates normally). PocketBase API calls fail silently with network errors — no UI error state is added in Phase 14 for offline data scenarios. Source: REQUIREMENTS.md PWA-ADV-01 deferred to v3.x.

## Copywriting Contract

All prior-phase copywriting is inherited unchanged. Phase 14 adds exactly two new strings.

| Element | Copy | Context |
|---------|------|---------|
| SW update toast message | "A new version of Wallecx is available." | `toast.info()` message — appears when a new SW is waiting to activate; duration: Infinity |
| SW update action button | "Refresh" | `action.label` — calls `updateServiceWorker(true)` and reloads |
| SW update cancel button | "Later" | `cancel.label` — dismisses toast; SW remains waiting |
| Session expired toast message | "Your session has expired. Please sign in again." | `toast.info()` message — appears on `onMounted` when `pb.authStore.isValid === false`; default duration (4000ms) |

**No new primary CTA** — this phase adds no new screen-level call-to-action.

**No new empty state** — this phase adds no new screens or views.

**No new destructive actions** — no confirmation dialogs are added in Phase 14. The "Refresh" button on the SW update toast is not destructive by convention (it is a system update action, not a data deletion).

**Copy rationale:**
- "A new version of Wallecx is available." — uses "Wallecx" (the app name, not "the app") for specificity. Present tense. Avoids technical language ("service worker", "update"). Source: 14-RESEARCH.md Pattern 3 (locked copy).
- "Refresh" — single word, matches browser-native update convention. Verb, not noun. Source: 14-RESEARCH.md Pattern 3.
- "Later" — single word, dismissive but not hostile. Does not say "Dismiss" or "Close" (implies permanence). Source: 14-RESEARCH.md Pattern 3.
- "Your session has expired. Please sign in again." — two sentences. First sentence describes the problem. Second sentence gives the exact action. Avoids "logout" (user did not actively log out). Source: 14-RESEARCH.md Pattern 3 (locked copy from REQUIREMENTS.md PWA-05).

## Constraint Checklist

Security and correctness constraints specific to Phase 14's UI surface.

| Constraint | Rule | Verifiable By |
|------------|------|---------------|
| `registerType: 'prompt'` — never `'autoUpdate'` | LOCKED in STATE.md. `autoUpdate` would silently reload the page and destroy unsaved CRUD form state. | `grep "registerType" vite.config.ts` — must show `'prompt'`, never `'autoUpdate'` |
| `duration: Infinity` on SW update toast | The toast must persist until user acts — no auto-dismiss. | Code review of `toast.info(...)` call in `WallecxApp.vue` watch handler |
| Session toast uses `toast.info()` not `toast.error()` | Session expiry is expected behavior; error severity is reserved for unexpected failures. | Code review of `onMounted` toast call |
| `pb.authStore.isValid` — not `useAuthStore().isLoggedIn` | `isLoggedIn` checks `!!user.value` (non-null record), not token expiry. `isValid` is the correct synchronous JWT expiry check. | Code review of `onMounted` branch condition |
| `navigator.storage.persist()` in optional-chain guard | `navigator.storage?.persist` — guards against iOS < 17 and older browsers. | Code review of `onMounted` — must use optional chaining |
| `toast()` before `router.push()` in session expiry flow | Toast must fire before the navigation so it is visible during the transition. | Code review of `onMounted` — `toast.info(...)` call precedes `await router.push(...)` |
| No `<link rel="manifest">` added manually to `index.html` | LOCKED in STATE.md. vite-plugin-pwa auto-injects it at build time. | `grep "rel=\"manifest\"" index.html` — must return zero hits |
| `scope: "/"` in manifest config | LOCKED in STATE.md. Sub-path scope breaks auth redirect UX in standalone mode. | `grep "scope" vite.config.ts` — must show `"/"` |
| `navigateFallback: 'index.html'` (no leading slash) | LOCKED in STATE.md. Without navigateFallback, offline SPA navigation returns a white screen. | `grep "navigateFallback" vite.config.ts` — must show `'index.html'` (no leading slash) |
| `devOptions.enabled: false` | SW must be disabled in development to avoid stale cache confusion during dev iteration. | `grep "devOptions" vite.config.ts` — `enabled: false` |
