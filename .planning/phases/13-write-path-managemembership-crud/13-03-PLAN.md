## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User → Delete confirm | Untrusted card_name interpolated into confirm message (plain text only — no v-html) |
| MembershipsTab → PocketBase delete | pb.collection.delete() call with server-enforced per-user rules |
| MembershipDetail emit → MembershipsTab handler | edit/delete events cross component boundary — no data payload, only intent signal |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-03-01 | Tampering | Delete confirm message — card_name injection | mitigate | D-11: message uses template literal interpolation in confirm.require() options object — not v-html. Vue's ConfirmDialog renders the message prop as text, not HTML. Verified in VaccinationsTab.vue analog. |
| T-13-03-02 | Tampering | Optimistic splice before server confirm | mitigate | Server-first delete pattern: await pb.delete() completes before splice; on catch, no splice occurs (tile stays visible) |
| T-13-03-03 | Elevation of Privilege | Cross-user delete via direct API call | transfer | PocketBase deleteRule enforces `@request.auth.id != "" && user = @request.auth.id` — server rejects any delete where auth.id != record.user (Phase 11 verified) |
| T-13-03-04 | Denial of Service | ConfirmDialog double-render from second instance | mitigate | Verified: ConfirmDialog NOT added to MembershipsTab.vue. useConfirm composable only. WallecxApp.vue provides the single instance. |
| T-13-03-05 | Tampering | Stale selectedRecord passed to openEdit on race condition | accept | Low risk: selectedRecord is set synchronously on tile click; openEdit closes detail before opening manage; no async gap where stale ref is a concern |
</threat_model>
