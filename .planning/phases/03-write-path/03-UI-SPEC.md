## Typography

Font family: Rubik, sans-serif — inherited from Phase 2. No new font sizes or weights introduced.

| Role | Size | Weight | Line Height | Token / Class | Used For |
|------|------|--------|-------------|---------------|----------|
| Body | 14px | 400 (Regular) | 1.5 | `text-sm` | Form field labels, field values in read states, action button labels, toast messages, ConfirmDialog message text, FileUpload label text, EXIF toast message |
| Heading | 24px | 700 (Bold) | 1.2 | `text-2xl font-bold` | WallecxApp page title "Wallecx" — unchanged from Phase 2 |

Notes:

- ConfirmDialog message: 14px regular body, rendered by PrimeVue Aura preset — no override.
- All user-supplied field values displayed in read states (DataTable rows, VaccinationDetail) remain at 14px weight 400 as per Phase 2 contract.

## Component Specifications

### 1. WallecxApp.vue — Modifications

**New state refs added:**
```
showManage: ref<boolean>(false)
manageRecord: ref<Vaccinations | null>(null)
```

**New handler functions:**
```
openManage(record: Vaccinations | null): void
  - manageRecord.value = record     // null = create mode; record = edit mode
  - showManage.value = true

onCreated(record: Vaccinations): void
  - records.value.unshift(record)   // newest first — matches sort order
  - (no toast here; ManageVaccination.vue fires the success toast)

onUpdated(updatedRecord: Vaccinations): void
  - const idx = records.value.findIndex(r => r.id === updatedRecord.id)
  - if (idx !== -1) records.value[idx] = updatedRecord
  - (no toast here; ManageVaccination.vue fires the success toast)

openDelete(record: Vaccinations): void
  - confirm.require({ ... })  // PrimeVue useConfirm service

deleteRecord(record: Vaccinations): Promise<void>
  - await pb.collection('wallecx_vaccinations').delete(record.id)
  - success: records.value.splice(idx, 1) → toast.success("Vaccination deleted.")
  - failure: toast.error("Failed to delete. Please try again.") — NO splice
```

**Explicit import required:**
```typescript
import { useConfirm } from "primevue/useconfirm";
```
(NOT auto-resolved by PrimeVueResolver — composable, not a component)

**Updated template structure:**
```
<Card>
```

### 5. Interaction States Summary

**"Add vaccination" header button (D-01):**
- Always visible in WallecxApp.vue header row (top-right), regardless of record count.
- `size="small"` with `icon="pi pi-plus"` prefix icon.
- Severity: PrimeVue default primary (navy fill, white label).
- Click → `openManage(null)` → ManageVaccination dialog opens in create mode.

**"Add your first vaccination" empty-state CTA (D-02):**
- Visible only when `records.length === 0` and `!isLoading`.
- `size="small"` with `icon="pi pi-plus"` prefix icon.
- Severity: PrimeVue default primary.
- Click → `emit('addFirst')` → WallecxApp calls `openManage(null)`.

**"Edit" row action:**
- `size="small"` `severity="secondary"` — removed `:disabled="true"` from Phase 2.
- Click → `emit('edit', data)` → WallecxApp calls `openManage(data)` → ManageVaccination opens in edit mode with initialValues populated.

**"Remove" row action:**
- `size="small"` `severity="danger"` — removed `:disabled="true"` from Phase 2.
- Click → `emit('remove', data)` → WallecxApp calls `openDelete(data)` → ConfirmDialog appears.

**ManageVaccination dialog — isSaving state:**
- `isSaving = true`: all `<InputText>`, `<InputNumber>`, `<DatePicker>`, `<Textarea>`, `<FileUpload>` disabled. Submit button shows loading spinner (`:loading="isSaving"`). Dialog X button disabled (`closable="!isSaving"`).
- `isSaving = false` (finally): controls re-enabled. Dialog may be closed by any mechanism.

**EXIF strip feedback (D-07):**
- After `onFileSelect` completes for an image: `toast.info("Location data removed.")`
- Fires unconditionally on every image upload — no GPS check required.
- PDFs: no toast.
- Fires before the form is submitted (during `@select` processing), not on form submit.

## Interaction & Motion

### Form — Validation

- `validate-on-submit` on the `<Form>` component — errors appear only after first submit attempt, not on blur.
- Inline `<Message severity="error" size="small" variant="simple">` renders directly below the field that fails.
- Multiple fields can show errors simultaneously if form submitted with multiple invalid fields.
- On correction + resubmit, resolved errors clear automatically.

### Form — Submit Flow

1. User clicks "Add Vaccination" / "Save Changes" button.
2. `isSaving.value = true` — button shows spinner, all controls disabled.
3. FormData assembled; PocketBase `create` or `update` called.
4. On success: `emit('created', record)` or `emit('updated', record)` → WallecxApp updates `records[]` → `toast.success(...)` → `visible.value = false` (dialog closes).
5. On failure: `toast.error(...)` — dialog stays open, `isSaving.value = false` (finally), user can retry.
6. Dialog fade-out: PrimeVue default CSS transition (~200ms).

### File Upload — Select Flow

1. User clicks FileUpload button → OS file picker opens.
2. `@select` fires with `{ files: File[] }` — use `event.files[0]` (PrimeVue v4 API).
3. MIME check: if rejected → `toast.error(...)` → return early. `pendingFile.value` not set.
4. Size check: if > 10 MB → `toast.error(...)` → return early.
5. PDF path: `pendingFile.value = file`. No toast.
6. Image path: canvas EXIF strip → `imageCompression()` (async, ~1–3s on mobile CPU) → `pendingFile.value = compressed` → `toast.info("Location data removed.")`.
7. `pendingFile.name` indicator appears below FileUpload (`"{filename} selected"`).

**Loading feedback during image compression:** No explicit spinner on the FileUpload element itself — compression runs in the background. The submit button is NOT disabled during file processing; only during the PocketBase request. This is intentional: user could fill other fields while image processes. If user submits before image finishes, `pendingFile.value` will be null (compression result not yet assigned) and the form will submit without the file — this is acceptable for v1.

### Delete — Confirm Flow

1. User clicks "Remove" in DataTable row.
2. `openDelete(record)` fires → `confirm.require(...)` called.
3. ConfirmDialog overlay appears (PrimeVue fade-in, ~200ms).
4. Focus lands on primary action per PrimeVue Aura default.
5. User clicks "Delete": `accept()` → `deleteRecord(record)` called.
6. On success: `records.value.splice(idx, 1)` → `toast.success("Vaccination deleted.")`. ConfirmDialog closes.
7. On failure: `toast.error("Failed to delete. Please try again.")` — no splice. ConfirmDialog closes (PrimeVue always closes after accept/reject).
8. User clicks "Keep Record": ConfirmDialog closes. No changes.
9. Escape or click-outside: PrimeVue ConfirmDialog default dismiss behavior — treated as reject (no delete).

### Toast Notifications — Phase 3 Summary

All toasts via `vue-sonner` `<Toaster />` in App.vue — unchanged mount, unchanged position (top-right), unchanged duration (4000ms auto-dismiss).

| Event | Toast Type | Message |
|-------|-----------|---------|
| Image upload processed | info | "Location data removed." |
| MIME validation fail | error | "File type not supported. Use JPEG, PNG, WebP, or PDF." |
| Size validation fail | error | "File too large. Maximum size is 10 MB." |
| Image processing error | error | "Failed to process image. Please try again." |
| Create success | success | "Vaccination added." |
| Update success | success | "Vaccination updated." |
| Save failure | error | "Failed to save. Please try again." |
| Delete success | success | "Vaccination deleted." |
| Delete failure | error | "Failed to delete. Please try again." |
