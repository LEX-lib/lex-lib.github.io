# Lexarium Requirements — v4.2 Budget Recovery & Hardening

**Milestone goal:** Eliminate the `404 Missing collection context` runtime error on the Expenses → Reports tab by ensuring the `wallecx_expense_budgets` PocketBase collection exists in production, and harden `ExpensesTab.vue` so a missing budgets collection no longer fires misleading toast copy or blocks expenses from loading.

**Context:** Discovered after v4.1 ship — Phase 28-01 Task 1 was a trust-based human-action checkpoint that never landed the collection in the production PocketBase instance. Surgical milestone; 2 phases.

## v4.2 Requirements

### Bug Recovery

- [x] **BUG-01**: `wallecx_expense_budgets` PocketBase collection exists in production with the locked Phase 28-01 schema (4 fields: user/category/budget_type/amount) and 5 per-user rules (createRule uses v0.29.3 `@request.body.user` syntax). A code-side smoke read returns 200 OK from the running app. Closed by Phase 31 via paste-back gated Admin UI flow + smoke verify (per the new D-13 invariant). Two accepted deviations recorded in 31-01-SUMMARY.md: (D-31-A) List/View/Update/Delete rules use the stricter `@request.auth.id != "" && user = @request.auth.id` form for project-wide consistency with other `wallecx_*` collections — accepted as strict enhancement; (D-31-B) literal `getList({ page: 1, perPage: 1 })` returns 400 due to a PocketBase v0.29.x count-path bug against non-trivial listRule expressions — functional equivalence verified via `getFullList()` returning 200 + empty array and `getList(p, pp, { skipTotal: true })` returning 200 + `items:[]`.

- [x] **BUG-02**: `ExpensesTab.vue` onMounted handles a missing or 404-returning `wallecx_expense_budgets` collection gracefully — expenses still load successfully even when the budgets fetch fails, the user sees an accurate toast (`'Failed to load budgets.'`) distinct from the expenses-load failure copy (`'Failed to load expenses. Pull to refresh or reload the page.'`), and the Reports tab degrades by silently rendering without the Budget vs Actual section. No misleading toast text. Closed by Phase 32 via surgical refactor: `loadBudgets()` gained `opts: { context: 'mount' | 'refresh' } = { context: 'refresh' }` parameter (ternary toast — mount copy verbatim from spec; save-refresh copy `'Failed to refresh budgets after save. Reload to see changes.'`); `onMounted` split into independent try/catches (`isLoading` wraps only expenses; budgets runs sequentially after). UAT 1 paste-back evidence: toast verbatim `'Failed to load budgets.'`, expenses list rendered, Reports opened, Budget vs Actual absent, console smoke probe returned 404 corroborating test exercise.

## Deferred Requirements (carried forward to v4.3+)

These were considered for v4.2 but explicitly deferred per user scoping decision — they are NOT gaps:

- **HEALTH-01** — Code-level collection health check on app boot or Reports tab mount: probe `wallecx_expense_budgets` reachability and surface an inline warning (with PB Admin UI setup steps) if missing. Prevents the same regression from re-occurring silently. Rationale to defer: the surgical fix in BUG-01 + BUG-02 unblocks the error; HEALTH-01 prevents the silent-regression failure mode in the future and is its own design effort. The new D-13 invariant in STATE.md (Admin-UI checkpoints require text paste-back + downstream smoke verify) already mitigates the trust-based-checkpoint failure mode at the workflow layer.
- **UAT-28-CLOSE** — Walk through Phase 28's 9 deferred UAT scenarios in `28-HUMAN-UAT.md` once the collection is healthy. Rationale to defer: collection now exists (BUG-01 closed); UAT walkthrough can happen in v4.3.
- **UAT-29-CLOSE** — Walk through Phase 29's 7 deferred UAT scenarios in `29-HUMAN-UAT.md` (period comparison validation). Rationale to defer: same as UAT-28-CLOSE; the period comparison line was observed rendering correctly during UAT 1 (`↑ ₱5,971.00 vs last month (no prior spend)` confirmed by user paste-back), but the full 7-scenario walkthrough is its own session.
- **PB-REALTIME** (newly captured during Phase 32 discussion) — PocketBase realtime `subscribe('*', cb)` / `unsubscribe('*')` for `wallecx_*` collections to eliminate the `@budgets-saved`-driven manual refetch pattern. Considered during Phase 32 discussion but rejected as scope creep — does not solve BUG-02 (the initial fetch still needs its own try/catch since subscribe does not deliver an initial snapshot), and ought to be applied uniformly across all wallecx collections rather than ad-hoc on one. Recommended for a uniform-application future phase.

## Out of Scope

| Item | Reason |
|------|--------|
| Schema-as-code / PocketBase migrations | PB instance has no migrations file in this repo; admin UI is the source of truth. Adopting migrations is its own larger initiative. |
| Auto-create missing collection from the client | Client-side cannot create collections; Admin API is admin-only. The fix is a one-time manual recreation, not a runtime auto-heal. |
| Decoupled fetches for vaccinations / memberships | Out of scope — those collections exist and load fine. Only the budgets fetch needs decoupling because it's the one that empirically fails. |
| Code-level collection health check | Deferred to v4.3 as HEALTH-01 (see Deferred Requirements above). |
| Phase 28 + 29 UAT walkthrough | Deferred to v4.3 as UAT-28-CLOSE + UAT-29-CLOSE (see Deferred Requirements above). |
