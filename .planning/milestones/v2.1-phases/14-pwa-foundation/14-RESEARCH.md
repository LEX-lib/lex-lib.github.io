## Summary

Phase 14 adds a complete PWA layer to the Lexarium SPA so that Wallecx is installable on iOS and Android home screens. The implementation centers on three tools: `vite-plugin-pwa` (service worker + manifest generation), `@vite-pwa/assets-generator` (icon generation from the existing `branding_logo.svg`), and a new `vercel.json` for cache headers.

The tech stack is highly compatible: vite-plugin-pwa 1.3.0 explicitly supports Vite 8 (the current version in use) as a peer dependency, and all required Workbox packages (`workbox-build@7.4.1`, `workbox-window@7.4.1`) are peer dependencies that npm will install. The `rolldownOptions` configuration already in `vite.config.ts` does not conflict with `VitePWA()` — the plugin operates at the Vite plugin level, not inside `rolldownOptions`.

**One scope conflict requires resolution before planning:** REQUIREMENTS.md (PWA-01) specifies `scope: "/projects/wallecx"` but STATE.md locked `scope: "/"`. These are mutually exclusive. The research section on Web App Manifest documents both tradeoffs so the planner can make the final call.

**Primary recommendation:** Add `vite-plugin-pwa` + `workbox-window` as devDependencies, configure with `generateSW` strategy, `registerType: 'prompt'`, `NetworkOnly` for the PocketBase domain, and `navigateFallback: 'index.html'`. Generate icons from `public/branding_logo.svg` using `@vite-pwa/assets-generator` with the `minimal-2023` preset. Create `vercel.json` from scratch with `Cache-Control: no-cache` for `sw.js` and `*.webmanifest`, plus the SPA catch-all rewrite. Add auth resilience and update toast to `WallecxApp.vue`.

## CRITICAL FLAG: Scope Conflict Between REQUIREMENTS.md and STATE.md

**REQUIREMENTS.md PWA-01** says: `scope: "/projects/wallecx"`
**STATE.md locked decision** says: `scope: "/"` — "Scoping to sub-path breaks cross-route navigation within the installed PWA."

These are contradictory. Here is the analysis:

| Scope Value | Behavior | Risk |
|-------------|----------|------|
| `scope: "/"` | All Lexarium routes stay in-scope when navigating from the installed PWA. No URL bar shown when going from `/projects/wallecx` to `/login`. | The installed PWA exposes the entire Lexarium site (not just Wallecx) under its scope. |
| `scope: "/projects/wallecx"` | Only `/projects/wallecx` is in-scope. Navigating to `/login` triggers the browser's out-of-scope indicator (URL bar appears in the standalone window). Tapping the router's login redirect looks like leaving the app. | Sub-path scope with an SPA login flow creates jarring UX. |

**State.md rationale is technically correct:** the router guard redirects expired sessions to `/login`, which is outside `/projects/wallecx`. With `scope: "/projects/wallecx"`, the installed PWA would show a browser URL bar on every auth redirect. This confirms STATE.md's `scope: "/"` is the better choice for user experience.

**Planner action required:** Use `scope: "/"` (STATE.md) and note the override of REQUIREMENTS.md PWA-01's scope field. The manifest `start_url` remains `/projects/wallecx`.

## Pattern 1: vite-plugin-pwa Configuration

**What:** VitePWA plugin added to `vite.config.ts`. Uses `generateSW` strategy (default). `registerType: 'prompt'` means the SW never auto-updates; it waits until the user confirms.

**When to use:** Single-page apps that cannot tolerate silent reloads (CRUD forms with unsaved state).

```typescript
// Source: Context7 /vite-pwa/vite-plugin-pwa + vite-pwa-org.netlify.app/deployment/vercel
// vite.config.ts

import { fileURLToPath, URL } from "node:url";
import { defineConfig } from "vite";
import vue from "@vitejs/plugin-vue";
import vueDevTools from "vite-plugin-vue-devtools";
import Components from "unplugin-vue-components/vite";
import { PrimeVueResolver } from "@primevue/auto-import-resolver";
import tailwindcss from "@tailwindcss/vite";
import { VitePWA } from "vite-plugin-pwa";

export default defineConfig({
  plugins: [
    vue({
      template: {
        compilerOptions: {
          isCustomElement: (tag) => tag === "iconify-icon",
        },
      },
    }),
    ...(process.env.NODE_ENV !== "production" ? [vueDevTools()] : []),
    Components({
      resolvers: [PrimeVueResolver()],
    }),
    tailwindcss(),
    VitePWA({
      registerType: "prompt",          // LOCKED: never 'autoUpdate'
      strategies: "generateSW",        // default; no custom sw.ts needed
      includeAssets: [
        "favicon.ico",
        "apple-touch-icon-180x180.png",
        "branding_logo.svg",
      ],
      manifest: {
        name: "Wallecx",
        short_name: "Wallecx",
        description: "Your personal vaccination and membership card vault",
        theme_color: "#002244",
        background_color: "#002244",
        display: "standalone",
        scope: "/",                    // LOCKED: not '/projects/wallecx' — see scope conflict section
        start_url: "/projects/wallecx",
        icons: [
          {
            src: "pwa-192x192.png",
            sizes: "192x192",
            type: "image/png",
            purpose: "any",
          },
          {
            src: "pwa-512x512.png",
            sizes: "512x512",
            type: "image/png",
            purpose: "any",
          },
          {
            src: "maskable-icon-512x512.png",
            sizes: "512x512",
            type: "image/png",
            purpose: "maskable",
          },
        ],
      },
      workbox: {
        // Precache all static assets — makes app shell available offline
        globPatterns: ["**/*.{js,css,html,ico,png,svg,woff2,webmanifest}"],
        cleanupOutdatedCaches: true,
        // navigateFallback is MANDATORY: SPA needs this for offline /projects/wallecx
        navigateFallback: "index.html",
        // Deny API and PocketBase URLs from navigateFallback (they are not navigation requests,
        // but belt-and-suspenders in case something slips through)
        navigateFallbackDenylist: [
          /^\/api\//,       // any future API routes
        ],
        // LOCKED: All PocketBase calls must be NetworkOnly
        // The PocketBase base URL is runtime-configured via VITE_API_BASE_URL.
        // Pattern matches any cross-origin request to the PocketBase host.
        runtimeCaching: [
          {
            // Match all requests to the PocketBase API origin
            // IMPORTANT: Use a broad pattern — the exact domain is injected at runtime,
            // but pocketbase.io or a self-hosted URL will not be same-origin with Vercel.
            // Cross-origin requests are NOT intercepted by the SW unless explicitly registered.
            // This entry is a safety net for same-origin API routes if any exist.
            urlPattern: /\/api\/.*/i,
            handler: "NetworkOnly",
          },
        ],
      },
      devOptions: {
        // Enable SW in development for local testing
        // Set to false if SW causes dev-mode confusion (it often does)
        enabled: false,
        type: "module",
      },
    }),
  ],
  resolve: {
    alias: {
      "@": fileURLToPath(new URL("./src", import.meta.url)),
    },
  },
  build: {
    outDir: "dist",
    rolldownOptions: {
      output: {
        codeSplitting: {
          groups: [
            { name: "leaflet", test: /\/leaflet/, priority: 30 },
            {
              name: "primevue",
              test: /\/primevue|\/@primevue|\/@primeuix/,
              priority: 20,
            },
            {
              name: "vendor",
              test: /\/vue|\/pinia|\/vue-router|\/@vue/,
              priority: 10,
            },
          ],
        },
      },
    },
  },
});
```

**Important note on PocketBase and NetworkOnly:** PocketBase is a cross-origin service (separate host from Vercel). Cross-origin requests from the browser do NOT pass through the service worker by default unless the service worker explicitly intercepts them via `runtimeCaching`. Since PocketBase is on a different origin (e.g., `https://your-pocketbase.pockethost.io`), its requests are NOT intercepted by the SW at all — they go directly to the network. The `NetworkOnly` `runtimeCaching` entry above is belt-and-suspenders for any same-origin API routes that might exist. The real protection is that the SW simply never sees cross-origin PocketBase requests.

## Pattern 3: SW Update Toast in WallecxApp.vue

**What:** `useRegisterSW` returns a reactive `needRefresh` ref that becomes `true` when a new SW is waiting. Watching it and calling `toast()` from vue-sonner creates a non-blocking update prompt. The `updateServiceWorker()` call triggers the waiting SW to activate and reloads the page.

**Critical constraint:** The Refresh action MUST NOT fire while a CRUD dialog is open. Both `MembershipsTab.vue` and `VaccinationsTab.vue` have their own local `showManage` and `showDetail` refs. `WallecxApp.vue` does not have access to these refs directly. The safe strategy is:

- **Do not disable the toast** based on dialog state (too complex, requires prop drilling or a store).
- **Do nothing special** — the user clicks "Refresh" voluntarily. If they click it while a form is open, they lose unsaved data. The toast is non-blocking; they can ignore it. The `registerType: 'prompt'` guarantee ensures no forced reload ever happens automatically.
- The toast should have `duration: Infinity` so it stays until dismissed — users are not rushed.

```typescript
// Source: Context7 /vite-pwa/vite-plugin-pwa + /xiaoluoboding/vue-sonner
// WallecxApp.vue <script setup lang="ts">

import { ref, onMounted, watch } from "vue";
import { useRouter } from "vue-router";
import { toast } from "vue-sonner";
import { useRegisterSW } from "virtual:pwa-register/vue";
import { pb } from "@/lib/pocketbase";
import VaccinationsTab from "./VaccinationsTab.vue";
import MembershipsTab from "./MembershipsTab.vue";

const router = useRouter();
const activeTab = ref<string>("vaccinations");

// --- PWA-06: SW update prompt ---
const { needRefresh, updateServiceWorker } = useRegisterSW();

watch(needRefresh, (val) => {
  if (!val) return;
  toast.info("A new version of Wallecx is available.", {
    duration: Infinity,
    action: {
      label: "Refresh",
      onClick: () => updateServiceWorker(true),
    },
    cancel: {
      label: "Later",
      onClick: () => {
        needRefresh.value = false;
      },
    },
  });
});

// --- PWA-05: Auth resilience + storage persistence ---
onMounted(async () => {
  // iOS 7-day localStorage eviction mitigation (does not guarantee persistence on iOS)
  if (navigator.storage?.persist) {
    try {
      await navigator.storage.persist();
    } catch {
      // Silently ignore — best-effort only
    }
  }

  // Check if PocketBase auth token is still valid
  // pb.authStore.isValid checks token expiry without a network request
  if (!pb.authStore.isValid) {
    toast.info("Your session has expired. Please sign in again.");
    await router.push({
      name: "login",
      query: { redirect: "/projects/wallecx" },
    });
  }
});
```

**Explanation of `pb.authStore.isValid`:** This is a synchronous getter on the PocketBase `AuthStore` that checks whether the stored JWT token is non-empty and not expired (by decoding the JWT expiry without a network round-trip). [VERIFIED: `node_modules/pocketbase/dist/pocketbase.cjs.d.ts` — `get isValid(): boolean`]

## Pattern 4: Icon Generation Command

**What:** `@vite-pwa/assets-generator` with the `minimal-2023` preset generates all required PWA icon sizes from a single source image. The source must be at least 512×512px (or an SVG).

**Source image situation:** `public/branding_logo.svg` is a complex branded SVG (1500×995 viewBox, landscape, with cream/navy colors). It will work as a source, but the resulting icons will be landscape-cropped, which may look odd as app icons. A square icon source (512×512 SVG with just the "C" letterform or logo mark) would produce better-looking icons. The planner should decide whether to:
1. Use `branding_logo.svg` as-is (fast, may look odd)
2. Create a `wallecx-icon.svg` square source specifically for PWA icons (better UX)

```bash
# Run from project root (generates files directly into public/)
npx @vite-pwa/assets-generator --preset minimal-2023 public/branding_logo.svg
```

**What `minimal-2023` preset generates:**
```
public/
├── favicon.ico                   (48×48, replaces existing)
├── pwa-64x64.png                 (transparent)
├── pwa-192x192.png               (transparent, purpose: "any")
├── pwa-512x512.png               (transparent, purpose: "any")
├── maskable-icon-512x512.png     (maskable, purpose: "maskable")
└── apple-touch-icon-180x180.png  (Apple touch icon)
```

[VERIFIED: Context7 /websites/vite-pwa-org_netlify_app — minimal-2023 preset definition]

**Caution:** The CLI overwrites `favicon.ico`. The current `index.html` references `branding_logo.svg` as the favicon (`<link rel="icon" type="image/svg+xml" href="/branding_logo.svg" />`). The generator will create a new `favicon.ico` but the `index.html` link to `branding_logo.svg` will remain valid (browsers prefer SVG over ICO when both are declared). The planner should decide whether to also update `index.html` to include the generated apple-touch-icon link.

**Note:** vite-plugin-pwa also supports an integrated `pwaAssets` config option that runs the generator during `vite build`. This is simpler but requires the source image to be ready at build time. Using the standalone CLI first (to verify icon quality) and then opting into `pwaAssets.config: true` later is the recommended two-step approach.

## Pattern 5: vercel.json (Complete File)

**What:** Vercel does not have a `vercel.json` in this project yet. The SPA is currently deployed without explicit header configuration, meaning Vercel applies its default caching. For `sw.js`, Vercel's default cache may persist the old SW indefinitely (CDN edge cache), preventing users from receiving SW updates. [CITED: vite-pwa-org.netlify.app/deployment/vercel]

```json
{
  "headers": [
    {
      "source": "/sw.js",
      "headers": [
        {
          "key": "Cache-Control",
          "value": "public, max-age=0, must-revalidate"
        }
      ]
    },
    {
      "source": "/(.*).webmanifest",
      "headers": [
        {
          "key": "Cache-Control",
          "value": "public, max-age=0, must-revalidate"
        },
        {
          "key": "Content-Type",
          "value": "application/manifest+json"
        }
      ]
    },
    {
      "source": "/(.*).html",
      "headers": [
        {
          "key": "Cache-Control",
          "value": "public, max-age=0, must-revalidate"
        }
      ]
    },
    {
      "source": "/assets/(.*)",
      "headers": [
        {
          "key": "Cache-Control",
          "value": "max-age=31536000, immutable"
        }
      ]
    }
  ],
  "rewrites": [
    {
      "source": "/(.*)",
      "destination": "/index.html"
    }
  ]
}
```

**Note on the SPA catch-all rewrite:** The `rewrites` rule sends all unmatched paths to `index.html`, which is required for Vue Router HTML5 history mode on Vercel. Without this, direct navigation to `/projects/wallecx` would return a 404. The existing GitHub Pages deploy (via `npm run deploy`) handles this by copying `dist/index.html` to `dist/404.html` (GitHub Pages convention). Vercel needs the `rewrites` rule instead. [ASSUMED: The existing Vercel deploy works without this rule because Vercel may already apply a default SPA rewrite — confirm by checking if direct URL navigation currently works in production. If it does, the rewrite may already be in a Vercel project setting.]

## Common Pitfalls

### Pitfall 1: `autoUpdate` in registerType

**What goes wrong:** Using `registerType: 'autoUpdate'` causes the new SW to silently skip waiting and reload the page immediately when a new build is deployed. Any user with an open ManageMembership or ManageVaccination dialog loses all unsaved form state.
**Why it happens:** `autoUpdate` sets `skipWaiting: true` and `clientsClaim: true` in the generated SW.
**How to avoid:** Always use `registerType: 'prompt'`. The `needRefresh` watch pattern gives users control.
**Warning signs:** If you see `skipWaiting: true` or `clientsClaim: true` in the generated `dist/sw.js`, the wrong registerType is configured.

### Pitfall 2: Missing vercel.json in Same Deployment as First SW

**What goes wrong:** If `sw.js` is deployed without `Cache-Control: no-cache`, Vercel's CDN edge may cache it for hours. Users who install the PWA will receive stale SW updates (or no updates) until the CDN cache expires.
**Why it happens:** Vercel applies CDN caching by default to all static files unless `vercel.json` overrides it.
**How to avoid:** Create `vercel.json` in the same PR/commit as the first `sw.js` deployment. STATE.md locked this requirement explicitly.
**Warning signs:** After deployment, check the response headers for `/sw.js` in the browser Network tab. If `Cache-Control` does not contain `max-age=0`, the vercel.json rules are not applying.

### Pitfall 3: `<link rel="manifest">` Added Manually in index.html

**What goes wrong:** If you add `<link rel="manifest" href="/manifest.webmanifest">` to `index.html`, it will conflict with vite-plugin-pwa's automatic injection, potentially pointing to the wrong file path or appearing twice.
**How to avoid:** STATE.md locked this: do NOT add it manually. vite-plugin-pwa injects it at build time.

### Pitfall 4: navigateFallback Missing or Wrong Path

**What goes wrong:** If `workbox.navigateFallback` is not set, or set to `'/index.html'` (with leading slash) when Vite's base is `/`, Workbox may not find the precached entry and return a network error for SPA navigation when offline.
**How to avoid:** Use `navigateFallback: 'index.html'` (no leading slash in the workbox config — Workbox will resolve it relative to the SW scope). Test offline by checking Chrome DevTools → Network → Offline mode.
**Warning signs:** Opening the installed PWA without network connectivity results in a blank white screen or generic error page instead of the cached app shell.

### Pitfall 5: `pb.authStore.isValid` vs `useAuthStore().isLoggedIn`

**What goes wrong:** Using `useAuthStore().isLoggedIn` for the mount check. The Pinia store's `isLoggedIn` is initialized from `pb.authStore.record` at store creation time (import time), not re-evaluated at `onMounted`. If the auth token expired during the 8-day iOS dormancy period, `pb.authStore.record` may be non-null but the token may be expired.
**Why it happens:** PocketBase's `authStore.record` stores the last-known user model in localStorage, even if the token has expired. The Pinia computed `isLoggedIn` checks `!!user.value`, not token validity.
**How to avoid:** Use `pb.authStore.isValid` directly in the `onMounted` check. This is a synchronous JWT expiry check that does not depend on the Pinia store reactivity initialization order.
**Warning signs:** An 8-day-dormant device opens the PWA, sees the Wallecx content briefly, then gets a 401 from PocketBase on the first data fetch.

### Pitfall 6: `scope: "/projects/wallecx"` Breaking Auth Redirect UX

**What goes wrong:** Setting manifest scope to `/projects/wallecx` causes the browser to display a URL bar (out-of-scope indicator) whenever the router guard redirects to `/login`. The installed PWA appears to "break" and open a browser window instead of staying in standalone mode.
**How to avoid:** Use `scope: "/"` per STATE.md's locked decision. The entire Lexarium SPA stays in-scope.
**Warning signs:** After installing the PWA and letting the session expire, re-opening the app shows a URL bar in what should be standalone mode.

### Pitfall 7: `branding_logo.svg` as Icon Source — Aspect Ratio

**What goes wrong:** `branding_logo.svg` has a landscape aspect ratio (1500×995). The `@vite-pwa/assets-generator` will crop or letterbox it when generating square icon PNGs. The result may be an unreadable icon on the home screen.
**Why it happens:** PWA icons are square. A landscape source SVG gets significant white space or is cropped awkwardly.
**How to avoid:** Create a dedicated square SVG icon source (e.g., `public/wallecx-icon.svg`) before running the generator. A simple navy (#002244) square with the "W" letterform or wallet icon is sufficient.
**Warning signs:** After running the generator, open `pwa-192x192.png` in an image viewer — if the icon content is very small or distorted, the source SVG needs adjusting.

### Pitfall 8: `sharp` Native Module on Windows

**What goes wrong:** `@vite-pwa/assets-generator` depends on `sharp`, which has native binaries. On Windows, installation may fail with platform-specific errors if the pre-built binary is not available for the exact Node version.
**Why it happens:** `sharp` uses `@img/sharp-win32-x64` for Windows x64. This pre-built binary must be available for Node 22.
**How to avoid:** Run `npm install -D @vite-pwa/assets-generator` and check for errors. If the binary install fails, use `--ignore-scripts` is NOT an option (sharp requires the binary). Alternative: run the generation step in WSL or in a CI pipeline.
**Warning signs:** `npm install` completes but `npx pwa-assets-generator` throws a module load error.

## navigator.storage.persist() Details

**Browser support as of 2026-05-14:**
- Chrome/Edge Android: Supported (auto-grants for installed PWAs)
- Safari iOS 17+: Supported (auto-grants based on interaction history, no user prompt)
- Safari iOS < 17: Not supported — graceful degradation via `navigator.storage?.persist`

[VERIFIED: WebSearch citing WebKit blog and MDN — Safari 17.0 added full Storage API support]

**iOS behavior:** On iOS, `navigator.storage.persist()` returns `true` if granted (iOS 17+) but the grant does not protect against Safari history clearing. Clearing Safari's browsing history on iOS removes ALL web storage including PWA storage, regardless of persistence grant. The `persist()` call specifically protects against the automatic 7-day eviction of inactive origins — which is the threat model for Wallecx (user has not visited in 8+ days, not user deliberately clearing history).

**Implementation pattern:**
```typescript
// Graceful degradation: optional chaining guards against iOS < 17 and older browsers
if (navigator.storage?.persist) {
  await navigator.storage.persist();
  // Returns Promise<boolean> — true if granted, false if denied
  // Do NOT show UI based on the return value — best-effort only
}
```

## Auth Resilience: Interaction with Existing Router Guard

The existing `router/index.ts` guard:
```typescript
router.beforeEach((to, from, next) => {
  const auth = useAuthStore();
  if (to.matched.some((record) => record.meta?.requiresAuth)) {
    if (!auth.isLoggedIn) {
      next({ name: "login", query: { redirect: to.fullPath } });
      return;
    }
  }
  next();
});
```

This guard uses `isLoggedIn` (Pinia computed from `pb.authStore.record`). It fires on every navigation but NOT on app mount if the user is already on `/projects/wallecx` (e.g., they bookmarked it or installed it as a PWA).

The `onMounted` check in `WallecxApp.vue` is complementary:
- **Router guard:** Catches navigation attempts when not logged in (including initial load via Vue Router).
- **onMounted check:** Catches the case where the user was previously logged in, the guard passed (record is non-null), but the token has expired (isValid is false). The router guard checks `!!user.value` (non-null record), not `pb.authStore.isValid` (token expiry). These are different conditions.

The `onMounted` check fires AFTER the router guard, so if the guard already redirected to `/login`, the component is unmounted before `onMounted` runs. The `onMounted` check only fires for the case where the guard allowed the navigation (record exists, isLoggedIn is true) but the token is actually expired.

**No conflict** between the two. Both are needed for robust coverage.
