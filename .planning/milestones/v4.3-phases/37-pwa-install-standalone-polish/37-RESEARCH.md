## Summary

Phase 37 completes the PWA experience for Wallecx by delivering six coordinated changes: iOS standalone meta tags in `index.html`, branded splash screens from `@vite-pwa/assets-generator`, an Android install button in the existing `PwaInstallBanner.vue`, safe-area polish for the SW-update toast, a new site-wide `OfflineBanner.vue` driven by `@vueuse/core` `useOnline`, and manifest `shortcuts` enabling Android long-press Quick Actions.

All primary implementation dependencies are already in place. `@vite-pwa/assets-generator@1.0.2` is already a devDependency — no new package install. `@vueuse/core` is a direct runtime dep since Phase 33. The `installPromptEvent` module singleton was wired in Phase 33 and awaits its first consumer. The `vite-plugin-pwa@1.3.0` manifest passthrough already supports arbitrary manifest keys including `shortcuts`.

The critical implementation risks are: (1) the `@vite-pwa/assets-generator` config file does not yet exist (`pwa-assets.config.ts` must be created at project root); (2) `public/splash/` and `public/shortcuts/` directories do not yet exist; (3) `vite.config.ts` `includeAssets` must be extended to reference the new PNG filenames so Workbox precaches them; (4) the `pendingAction` dispatch in WallecxApp.vue requires careful ordering to avoid firing before the relevant tab component's watcher is registered.

**Primary recommendation:** Implement in five sequential waves: (1) index.html iOS meta tags + theme-color, (2) @vite-pwa/assets-generator config + splash generation + index.html link tags, (3) PwaInstallBanner.vue Android branch + dismissal schema migration, (4) OfflineBanner.vue + App.vue mount + ROADMAP/REQUIREMENTS reword, (5) manifest shortcuts + vite.config.ts extension + WallecxApp pendingAction dispatch + guard spec + SW toast safe-area.

### Pattern 1: @vite-pwa/assets-generator Config File

**What:** `pwa-assets.config.ts` at project root drives the CLI. Uses `combinePresetAndAppleSplashScreens` from the config entrypoint to attach splash generation to the `minimal2023Preset`.

**When to use:** Required once to generate splash PNGs and shortcut icons. Run via `npx @vite-pwa/assets-generator` (the CLI reads the config file automatically).

**Key facts** [CITED: https://vite-pwa-org.netlify.app/assets-generator/cli]:
- Config file: `pwa-assets.config.ts` at project root
- Entrypoint: `@vite-pwa/assets-generator/config`
- Splash helper: `combinePresetAndAppleSplashScreens(preset, splashOptions, deviceNames?)`
- If `deviceNames` is omitted, splashes are generated for ALL Apple devices
- Background color is set via `resizeOptions.background`
- The CLI also accepts `--preset minimal-2023 <image>` for zero-config runs

**Example:**
```typescript
// Source: https://vite-pwa-org.netlify.app/assets-generator/cli
// pwa-assets.config.ts
import {
  defineConfig,
  minimal2023Preset,
  combinePresetAndAppleSplashScreens
} from '@vite-pwa/assets-generator/config'

export default defineConfig({
  headLinkOptions: {
    preset: '2023'
  },
  preset: combinePresetAndAppleSplashScreens(
    minimal2023Preset,
    {
      padding: 0.3,
      resizeOptions: { background: '#002244', fit: 'contain' },
      linkMediaOptions: { log: true, basePath: '/' }
    }
    // Omit device list to generate for all Apple devices (then copy 3 needed PNGs)
    // OR pass: ['iPhone 14 Pro', 'iPad Air 9.7"'] if those exact strings are valid
  ),
  images: ['public/branding_logo.svg']
})
```

> **Important:** The assets-generator outputs icon files TO THE SAME DIRECTORY as the source image by default. For splash screens in a subdirectory (`public/splash/`), either move the output manually after generation or configure the output path. The exact CLI `--output` option availability needs to be verified during implementation — if unavailable, generate to `public/` and move files to `public/splash/` as a post-generation step. [ASSUMED] regarding output path configurability.

### Pattern 2: BeforeInstallPromptEvent.prompt() Single-Use Pattern

**What:** Call `.prompt()` once on the captured event, await `.userChoice`, handle `accepted`/`dismissed`, clear the singleton. The event is single-use — never call `.prompt()` twice on the same reference.

**When to use:** Inside the Install button click handler in PwaInstallBanner.vue Android branch.

**Example:**
```typescript
// Source: https://web.dev/learn/pwa/installation-prompt [CITED]
// PwaInstallBanner.vue Android branch handler
import { useMobileEnv, clearInstallPromptEvent } from '@/composables/useMobileEnv'

const { installPromptEvent, isStandalone } = useMobileEnv()

async function handleAndroidInstall(): Promise<void> {
  const event = installPromptEvent.value
  if (!event) return
  
  event.prompt()  // Show browser install dialog (must be in user-gesture handler)
  
  const { outcome } = await event.userChoice  // Wait for user choice
  
  if (outcome === 'dismissed') {
    // Write 30-day dismissal record (D-37-04)
    writeDismissalRecord('android')
  }
  // Both outcomes: clear the singleton so banner hides for this session
  clearInstallPromptEvent()
}
```

**Critical constraints:**
- `prompt()` MUST be called from a user-gesture handler (click). Calling it asynchronously (after an await) may break the user-gesture requirement in some browsers. [ASSUMED: the await on userChoice may be safe since prompt() itself is called synchronously in the click handler, but verify on real device.]
- After `prompt()`, the event object is consumed. `clearInstallPromptEvent()` prevents any re-call.
- Chrome fires `appinstalled` on install acceptance, which App.vue already handles by calling `clearInstallPromptEvent()` — this is belt-and-suspenders.

### Pattern 5: pendingAction Dispatch in WallecxApp.vue

**What:** Read `route.query.action` on mount, map to `activeTab` + set `pendingAction` ref, then clear the query param. Tab components watch `pendingAction` to open dialogs.

**Critical ordering:** The `route.query.action` read must happen AFTER the Suspense-wrapped async tab components have resolved. If `pendingAction` is set before `ExpensesTab`'s watcher is registered, the `add-expense` action would be missed.

**Recommended approach:**
```typescript
// WallecxApp.vue — inside the existing onMounted block
// After the storage.persist + auth check logic:
const ACTION_TAB_MAP: Record<string, string> = {
  'add-expense': 'expenses',
  'add-vaccination': 'vaccinations',
  'add-membership': 'memberships',
  'open-reports': 'expenses',
}

const pendingAction = ref<string | null>(null)

onMounted(async () => {
  // ... existing persist + auth check ...
  
  const action = route.query.action as string | undefined
  if (action && ACTION_TAB_MAP[action]) {
    activeTab.value = ACTION_TAB_MAP[action]
    // Use nextTick to defer pendingAction assignment until after tab renders
    await nextTick()
    pendingAction.value = action
    router.replace({ query: {} })
  }
})
```

**Tab-side watcher** (example for ExpensesTab):
```typescript
// ExpensesTab.vue — receives pendingAction as prop or reads from parent
// OR WallecxApp.vue passes it as a prop to async components
watch(() => props.pendingAction, (action) => {
  if (action === 'add-expense') openManageExpense()
  if (action === 'open-reports') activeSubTab.value = 'reports'
})
```

> **Planner decision required:** Choose whether `pendingAction` is passed as a prop to the tab components, or whether each tab component calls `useMobileEnv()`-like pattern to access a module-level ref. The locked invariant (no new Pinia store) means either a prop or a module-scope ref (similar to `installPromptEvent`) is acceptable. A prop is simpler and avoids a new module-scope singleton.

### Pattern 7: SW-Update Toast Safe-Area

**What:** vue-sonner's `toast.info()` accepts a `style` object in its options. Alternatively, the global `<Toaster />` component accepts `toastOptions`. A third option is a CSS override targeting `[data-sonner-toaster]`.

**Planner decision:** The simplest and most targeted approach is to pass `style: { paddingBottom: 'env(safe-area-inset-bottom)' }` directly in the existing `toast.info(...)` call at WallecxApp.vue lines 25–37. This does not affect other toasts site-wide (e.g., the auth-expired toast). If ALL toasts need safe-area treatment, use the `toastOptions` prop on `<Toaster />` in App.vue.

**Example (per-call approach — targeted):**

// Source: https://github.com/xiaoluoboding/vue-sonner [CITED]

  duration: Infinity,
  style: { paddingBottom: 'env(safe-area-inset-bottom)' },  // PWA-06 safe-area

**CSS override alternative (global):**
```css
/* wallecx-overrides.css */
[data-sonner-toaster] {

```

### Anti-Patterns to Avoid

- **Calling `prompt()` twice on the same event reference** — the event is single-use; calling `prompt()` a second time is a no-op in Chrome and may throw in other browsers. Always `clearInstallPromptEvent()` after the first call. (M-9 invariant)
- **Checking `installPromptEvent.value` for the iOS branch** — iOS Safari NEVER fires `beforeinstallprompt`; iOS branch must use UA detection (`isIosSafari()`). Checking the event ref on iOS will always see `null`.
- **Adding `registerType: 'autoUpdate'` or narrowing `scope`** — both are LOCKED; any edit to those lines breaks NFR-PWA-AUTOUPDATE and CON-PWA-SCOPE.
- **Adding `<link rel="manifest">` to index.html manually** — vite-plugin-pwa auto-injects it at build time; a manual entry creates a duplicate.
- **Setting `paddingBottom` on sticky bottom banners without the `+0.75rem` minimum** — `env(safe-area-inset-bottom)` resolves to `0` on non-notched devices, collapsing the bottom margin entirely. Always use `calc(env(safe-area-inset-bottom) + 0.75rem)`.
- **Setting `paddingTop` on the offline banner without `env(safe-area-inset-top)`** — the top-fixed offline banner will overlap the iOS status bar on notched/dynamic-island devices.
- **Teleporting OfflineBanner to `body` without a high z-index** — the offline banner competes with CustomNavBar (which has `paddingTop: env(safe-area-inset-top)`) and PrimeVue overlays. `z-index: 50` matches the install banner; ensure this is above the navbar but below modals.

## Common Pitfalls

### Pitfall 1: `prompt()` Called Outside User-Gesture Context

**What goes wrong:** Chrome requires `prompt()` to be called synchronously within a user-gesture handler. If an `await` precedes the `prompt()` call, the browser may reject it as not-user-initiated.

**Why it happens:** The template calls `await handleAndroidInstall()` but `prompt()` is inside the async function after an await.

**How to avoid:** Call `event.prompt()` FIRST in the click handler (synchronously), then `await event.userChoice`. The pattern is: `event.prompt()` → `const { outcome } = await event.userChoice` — not the other way around. [VERIFIED: web.dev/learn/pwa/installation-prompt]

**Warning signs:** Chrome DevTools console shows "The page attempted to prompt for install, but Chrome requires a user gesture" after the prompt() call.

---

### Pitfall 2: iOS Banner Shows in Standalone Mode

**What goes wrong:** If `isInStandaloneMode()` check is missing from the show logic, the iOS install instructions show even when the app is already installed as a PWA.

**Why it happens:** `navigator.standalone === true` is only set when running as a PWA on iOS. If the check uses only `installPromptEvent === null` (which is always null on iOS), it misses the standalone suppression.

**How to avoid:** Both branches must check `isStandalone` (from `useMobileEnv`) as the FIRST gate: `if (isStandalone.value) return` before any other show logic. The `isInStandaloneMode()` local function in `PwaInstallBanner.vue` already implements this correctly — DO NOT remove it when adding the Android branch.

---

### Pitfall 3: Legacy Dismissal Record Always Triggers Re-show

**What goes wrong:** If the migration from `'true'` → `{ dismissedAt }` is not implemented, all existing iOS users who previously dismissed the banner (value is `'true'`) will see the banner immediately after Phase 37 deploys, because `JSON.parse('true')` succeeds as a boolean and `Date.parse(true)` returns `NaN`, making `Date.now() - NaN >= threshold` evaluate as `false`, so the gate passes and the banner shows.

**How to avoid:** Implement D-37-07 migration: on read, if the stored value is the string `'true'` (not JSON), parse fails → write `{ dismissedAt: new Date().toISOString(), platform: detected }`. Only THEN apply the 30-day gate.

**Warning signs:** Banner appears immediately on install-fresh devices with prior dismissal records.

---

### Pitfall 4: pendingAction Fires Before Tab Watcher is Registered

**What goes wrong:** `pendingAction.value = 'add-expense'` is set in `onMounted` before the async `ExpensesTab` component resolves from its `Suspense` wrapper. The watcher in `ExpensesTab` hasn't registered yet, so the action is missed silently.

**How to avoid:** Use `await nextTick()` after setting `activeTab.value` but BEFORE setting `pendingAction.value`. This gives Vue time to begin resolving the Suspense for the active tab. Additionally, the tab component's watcher should use `{ immediate: true }` to catch `pendingAction` values already set when the watcher registers.

**Warning signs:** Shortcut "Add Expense" navigates to the Expenses tab but does NOT open the ManageExpense dialog.

---

### Pitfall 5: Workbox 3 MiB Limit with New Splash PNGs

**What goes wrong:** Splash screen PNGs for high-DPI devices can be large (1179×2556 @3x = potentially 500 KB+ each). If any individual PNG exceeds the 3 MiB `maximumFileSizeToCacheInBytes` limit, Workbox will skip it with "Skipping precaching" in the build log, and the PWA won't work offline.

**How to avoid:** After generating splash PNGs, run `npm run build` and inspect the precache log for "Skipping precaching" or "exceeds" messages (NFR-PWA-PRECACHE-FITS). The `@vite-pwa/assets-generator` uses `compressionLevel: 9, quality: 60` by default — splash PNGs should be well under 3 MiB each. However, if a splash exceeds the limit, increase the Workbox `maximumFileSizeToCacheInBytes` or exclude splashes from precache via `globIgnores`.

**Warning signs:** `npm run build` output contains "Skipping precaching of public/splash/apple-splash-*.png".

---

### Pitfall 6: OfflineBanner z-index Collision with CustomNavBar

**What goes wrong:** `OfflineBanner` is fixed to the top of the viewport. `CustomNavBar` is also at the top, styled with `paddingTop: env(safe-area-inset-top)`. If z-index ordering is wrong, the offline banner renders BEHIND the navbar.

**How to avoid:** Mount `OfflineBanner` in `App.vue` BEFORE `<CustomNavBar>` in template order (or ensure z-index 50+ on the banner). Teleport to body ensures stacking context independence. Confirm `z-index: 50` (same as install banner) is high enough above PrimeVue component defaults.

**Warning signs:** Offline banner appears invisible when triggered; inspecting DOM shows it is rendered but z-ordered behind navbar.

---

### Pitfall 7: BarcodeDisplay Regression from theme-color / iOS CSS

**What goes wrong:** Adding `apple-mobile-web-app-status-bar-style="black-translucent"` changes the iOS status-bar rendering. If any CSS cascade for standalone mode changes background colors globally, `BarcodeDisplay.vue`'s hardcoded `bg-white` / `background: white` might be overridden.

**How to avoid:** `BarcodeDisplay.vue` uses Tailwind `bg-white` and inline `background: 'FFFF00'` (via `BARCODE_BACKGROUND` constant — actually white from codebase inspection). These are hardcoded, not CSS-variable-driven. Theme-color meta tags and standalone CSS additions should NOT affect hardcoded color utilities. Verify with an eyeball check: open BarcodeDisplay in light mode, dark mode, and standalone after Phase 37 changes land.

### PwaInstallBanner.vue — Updated Dismissal Schema (D-37-06/D-37-07)

```typescript
// Source: 37-CONTEXT.md D-37-06, D-37-07 [CITED]
// PwaInstallBanner.vue — replace the existing onMounted dismissal check

interface DismissalRecord {
  dismissedAt: string  // ISO8601
  platform: 'ios' | 'android'
}

const THIRTY_DAYS_MS = 30 * 24 * 60 * 60 * 1000

function isDismissed(): boolean {
  try {
    const raw = localStorage.getItem(BANNER_DISMISSED_KEY)
    if (!raw) return false
    
    // D-37-07: lazy migration of Phase 14 schema
    if (raw === 'true') {
      const migrated: DismissalRecord = {
        dismissedAt: new Date().toISOString(),
        platform: isIosSafari() ? 'ios' : 'android'
      }
      localStorage.setItem(BANNER_DISMISSED_KEY, JSON.stringify(migrated))
      return true  // 30-day clock starts now
    }
    
    const parsed: DismissalRecord = JSON.parse(raw)
    return Date.now() - Date.parse(parsed.dismissedAt) < THIRTY_DAYS_MS
  } catch {
    return true  // localStorage failure → suppress banner
  }
}

function writeDismissalRecord(platform: 'ios' | 'android'): void {
  try {
    const record: DismissalRecord = {
      dismissedAt: new Date().toISOString(),
      platform
    }
    localStorage.setItem(BANNER_DISMISSED_KEY, JSON.stringify(record))
  } catch {
    // Degrade silently
  }
}
```
