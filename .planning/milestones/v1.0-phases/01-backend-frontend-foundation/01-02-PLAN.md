  <what-to-do>
    Perform the following steps using two real PocketBase user accounts. The existing API Playground at `/projects/api-playground` can run tests 1–4 directly without writing any code.

    **Setup: Create User A's test record**

    1. Log in to the Lexarium app (`npm run dev` → `/login`) as your regular developer account (User A).
    2. Open the PocketBase Admin UI. Navigate to `wallecx_vaccinations`. Click "New record".
    3. Fill in:
       - user: (select your User A account)
       - vaccine_name: `Smoke Test Vaccine`
       - date_administered: any date, e.g. `2024-01-15 00:00:00.000Z`
       - Optionally attach any file (PNG, JPG, or PDF) to the `card` field.
    4. Save the record. Note:
       - The record's `id` (e.g. `abc123xyz`)
       - The `card` file URL, which appears in the Admin UI as something like:
         `http://127.0.0.1:8090/api/files/wallecx_vaccinations/abc123xyz/filename.png`

    **Setup: Create User B**

    5. In PocketBase Admin UI, go to "Users" (or `_pb_users_auth_` collection). Create a second user account — use any email/password (e.g. `userb@test.local` / `testpassword123`).
    6. Log in to the Lexarium app as User B (open an incognito/private window, navigate to `/login`, sign in with User B's credentials).
    7. Note User B's auth token — in the API Playground at `/projects/api-playground`, set the Authorization header to `Bearer <userB_token>` for tests 1–4. (User B's token is returned by the login response.)

    **Test 1: List isolation**

    Request: `GET http://127.0.0.1:8090/api/collections/wallecx_vaccinations/records`
    Auth: User B's token in Authorization header.
    Expected: HTTP 200 with `items: []` (empty array — User B sees zero records).
    FAIL if: `items` contains any records belonging to User A.

    **Test 2: View isolation**

    Request: `GET http://127.0.0.1:8090/api/collections/wallecx_vaccinations/records/{userA_record_id}`
    Auth: User B's token.
    Expected: HTTP 404 (record not found — collection rule treats it as non-existent for User B).
    FAIL if: HTTP 200 is returned with User A's record data.

    **Test 3: Update isolation**

    Request: `PATCH http://127.0.0.1:8090/api/collections/wallecx_vaccinations/records/{userA_record_id}`
    Body: `{ "vaccine_name": "HACKED" }`
    Auth: User B's token.
    Expected: HTTP 403 or 404.
    FAIL if: HTTP 200 is returned and the record is modified.

    **Test 4: Delete isolation**

    Request: `DELETE http://127.0.0.1:8090/api/collections/wallecx_vaccinations/records/{userA_record_id}`
    Auth: User B's token.
    Expected: HTTP 403 or 404.
    FAIL if: HTTP 200 or 204 is returned and the record is deleted.

    **Test 5: File URL isolation**

    Action: Copy the card file URL from Step 4 above (e.g. `http://127.0.0.1:8090/api/files/wallecx_vaccinations/abc123xyz/filename.png`). Open this URL in a fresh incognito/private browser tab where you are NOT logged in (no PocketBase session cookie).
    Expected: HTTP 403 response (because `protected: true` requires a short-lived token).
    FAIL if: The file downloads or displays successfully without a token.

    **If any test fails:**

    - Recheck the collection rules in Task 1 of Plan 01 — copy the exact rule strings and save again.
    - Verify the `card` field has `protected: true` enabled.
    - Run the failing test again after each fix.
    - Do NOT proceed to Plan 03 until all five tests pass.

    **Cleanup (optional but recommended):**

    After all five tests pass, delete the smoke test record from the Admin UI (User A's `Smoke Test Vaccine` record). User B's account can remain in PocketBase for future testing.
  </what-to-do>
