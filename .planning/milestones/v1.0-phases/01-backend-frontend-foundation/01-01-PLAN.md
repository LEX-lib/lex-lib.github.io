    In the `wallecx_vaccinations` collection editor, click the "API Rules" tab (sometimes labelled "Rules" or "Access").

    **Step 2: Set each rule — copy these strings EXACTLY (no leading/trailing spaces)**

    List rule:
    ```
    @request.auth.id != "" && user = @request.auth.id
    ```

    View rule:
    ```
    @request.auth.id != "" && user = @request.auth.id
    ```

    Update rule:
    ```
    @request.auth.id != "" && user = @request.auth.id
    ```

    Delete rule:
    ```
    @request.auth.id != "" && user = @request.auth.id
    ```

    Create rule (DIFFERENT — uses @request.data.user, not user):
    ```
    @request.auth.id != "" && @request.data.user = @request.auth.id
    ```

    CRITICAL: The create rule uses `@request.data.user`, not `user`. `@request.data.*` is the correct form for submitted body parameters in PocketBase v0.26.x. Do NOT use `@request.body.user` — that syntax does not exist in v0.26.x and will cause all create requests to return 403.

    **Step 3: Save the rules**
    Click "Save" or "Save rules" after entering all five rules.

    **Step 4: Add the composite index (BACK-04)**

    Option A — Admin UI Indexes section (preferred if visible):
    - In the collection editor, look for an "Indexes" tab or section.
    - Click "Add index" or the "+" button in the Indexes section.
    - Paste the following SQL exactly:
      ```sql
      CREATE INDEX idx_wallecx_vaccinations_user_date ON wallecx_vaccinations (user, date_administered)
      ```
    - Click "Save" or "Add".

    Option B — JS migration fallback (if Admin UI has no Indexes section):
    - Create the file `pb_migrations/TIMESTAMP_add_wallecx_vaccinations_index.js` (replace TIMESTAMP with the current Unix timestamp, e.g. `1715356800`):
      ```javascript
      migrate(
        (app) => {
          const collection = app.findCollectionByNameOrId("wallecx_vaccinations");
          collection.indexes.push(
            "CREATE INDEX idx_wallecx_vaccinations_user_date ON wallecx_vaccinations (user, date_administered)"
          );
          app.save(collection);
        },
        (app) => {
          const collection = app.findCollectionByNameOrId("wallecx_vaccinations");
          collection.indexes = collection.indexes.filter(
            (i) => !i.includes("idx_wallecx_vaccinations_user_date")
          );
          app.save(collection);
        }
      );
      ```
    - PocketBase auto-runs JS migrations in `pb_migrations/` on startup. Restart PocketBase after creating the file.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Unauthenticated request → PocketBase API | Rules must reject any request without a valid auth token |
| Authenticated user → other users' records | Rules must enforce user = @request.auth.id so User B cannot access User A's records |
| Uploaded file URL → browser without token | protected: true forces token requirement; prevents URL-guessing attacks |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Information Disclosure | wallecx_vaccinations listRule | mitigate | listRule `@request.auth.id != "" && user = @request.auth.id` ensures only the record owner sees their own records; unauthenticated returns 403 |
| T-01-02 | Elevation of Privilege | wallecx_vaccinations createRule | mitigate | createRule `@request.data.user = @request.auth.id` prevents a user from creating records assigned to another user's id; reject if payload user != auth id |
| T-01-03 | Information Disclosure | card file field without protected: true | mitigate | protected: true on file field requires short-lived token for file access; direct URL without token returns 403 |
| T-01-04 | Tampering | wallecx_vaccinations updateRule | mitigate | updateRule `user = @request.auth.id` prevents User B from patching User A's records even with a known record id |
| T-01-05 | Tampering | wallecx_vaccinations deleteRule | mitigate | deleteRule `user = @request.auth.id` prevents User B from deleting User A's records |
| T-01-06 | Information Disclosure | notes field as EditorField | accept | notes is a plain TextField; EditorField (which stores HTML) is explicitly avoided to prevent v-html injection in Phase 2 |
</threat_model>
