## Overview

Surgical recovery milestone to close two production-visible bugs discovered after v4.1 ship:

- **BUG-01** — Phase 28-01's `wallecx_expense_budgets` collection was never landed in production PocketBase because Task 1 was a trust-based human-action checkpoint with no code-level verification (the silent-no-op failure mode). The Expenses → Reports tab fired `ClientResponseError 404: Missing collection context` on every mount.
- **BUG-02** — The shared try/catch in `ExpensesTab.vue` `onMounted` surfaced the BUG-01 404 as a misleading `'Failed to load expenses…'` toast and kept the expenses skeleton on screen even when expenses themselves loaded fine. Root cause: one try/catch wrapping two independent PocketBase calls.

The milestone landed the collection via a paste-back gated Admin UI flow, refactored `ExpensesTab.vue` to isolate the budgets path, and locked a new architectural invariant (D-13) that prevents the trust-based-checkpoint silent-no-op failure mode project-wide.

## Milestone Summary

**Decimal Phases:** None — Phase 31 and Phase 32 are straight-numbered phases scoped via `/gsd-new-milestone` after v4.1 shipped.

**Key Decisions:**

- **D-13: Admin-UI checkpoints require text paste-back + downstream smoke verify.** Locked into STATE.md `## Architectural Invariants` during Phase 31 Task 3 Part C. Now binds all future GSD admin-UI tasks in this project. Acknowledgment-only signals ("approved", "done", "verified") are explicitly insufficient — actual configured values must be pasted as text AND a code-side smoke verification must exercise the live artifact. Eliminates the silent-no-op failure mode at the workflow layer, not just per-phase.
- **D-31-A (Phase 31): Stricter listRule variant accepted with rationale.** Spec said `user = @request.auth.id`; user configured `@request.auth.id != "" && user = @request.auth.id` for project-wide consistency with other `wallecx_*` collections. Strict enhancement (explicit unauthenticated-rejection prefix), no security regression. The paste-back gate's literal-match policy was respected by surfacing the diff and recording the waiver explicitly rather than silently accepting.
- **D-31-B (Phase 31): PocketBase v0.29.x count-path bug discovered + documented.** `getList()` against collections with non-trivial listRule expressions (`@request.auth.id != "" && user = @request.auth.id`) returns 400 in the totalItems count path. Workarounds: `getFullList()` (uses skipTotal internally) or `getList(p, pp, { skipTotal: true })`. Diagnosed during Phase 31 Task 3 Part B via three independent probes. Functional-equivalence waiver accepted because the underlying semantic (collection accessible, zero rows match for authenticated user) was proven three ways.
- **D-32-1 (Phase 32): `loadBudgets()` gains parameterized `context`.** Signature: `async function loadBudgets(opts: { context: 'mount' | 'refresh' } = { context: 'refresh' }): Promise<void>`. Default `'refresh'` preserves the existing `@budgets-saved="loadBudgets"` template binding without modification (verified: `ExpensesReportsView.vue` line 30 emits `'budgets-saved': []` no payload). Single source of truth; zero duplication; distinct toast copy per call site.
- **D-32-2 (Phase 32): Toast copy differentiation.** Mount-time: `'Failed to load budgets.'` (ROADMAP-locked verbatim); save-refresh: `'Failed to refresh budgets after save. Reload to see changes.'` (replaces the prior ambiguous `'Failed to refresh budgets. Please reload the page.'` — explicit post-save context so the user understands their save may have landed but the in-memory view is stale).
- **D-32-3 (Phase 32): `isLoading` wraps expenses only.** The expenses fetch sits inside `try/finally`; `loadBudgets({ context: 'mount' })` runs sequentially after — NOT inside the `isLoading` envelope. Matches BUG-02 isolation intent: no part of the budgets data path can affect the expenses UX.
