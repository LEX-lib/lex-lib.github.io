## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| sessionStorage (same-origin) → viewMode ref | Other code on the same origin (now or in future) could write arbitrary values to the `wallecx:view-mode` key |
| viewMode ref → DOM class binding | The viewMode value flows into the `gridClass` computed which is rendered as a class attribute |
| viewMode ref → WallecxToolbar prop | The viewMode value is passed to the toolbar's icon/aria/title ternaries (Plan 1 handles those bindings) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-05 | Tampering | sessionStorage hydration in `onMounted` — untrusted value injected by other same-origin code into `wallecx:view-mode` | mitigate | The hydration path validates the stored value with the strict literal disjunction `stored === 'grid' \|\| stored === 'list'` BEFORE assigning to `viewMode.value`. Any other value (null, an empty string, JSON garbage, an injected payload like `'<script>'` or `'{__proto__: ...}'`) is silently rejected and the default `'grid'` is kept. The two accepted values are then used only to select between hardcoded Tailwind class strings inside `gridClass`, so even valid values cannot escape into a wider attack surface. Acceptance criteria explicitly grep-checks for the literal guard string. |
| T-08-06 | Denial of Service | sessionStorage I/O — privacy-mode iframes / quota-exceeded browsers throw on `getItem`/`setItem` | mitigate | Both the read in `onMounted` and the write in `watch(viewMode)` are wrapped in `try { ... } catch { ... }` with empty catch bodies. A throw on read keeps the default `'grid'`. A throw on write means this session simply doesn't persist — no toast, no console.error, no crash. The component remains fully functional without storage. |
| T-08-07 | Injection (XSS) | `:class="gridClass"` binding on the grouped-cards `<div v-else>` | accept | `gridClass` is a `computed` that returns ONE of two hardcoded string literals — `'grid grid-cols-1 gap-4'` or `'grid grid-cols-1 sm:grid-cols-2 gap-4'`. Acceptance criteria grep-checks for absence of `v-html`. |
| T-08-08 | Information Disclosure | sessionStorage write — does the stored value leak across users? | accept | sessionStorage is scoped per-tab and per-origin. The value is the literal string `'grid'` or `'list'` — non-sensitive UI state. No PII, no auth tokens, no record content. A multi-user shared-browser scenario would result only in the previous user's preferred layout persisting until they navigate away or close the tab — a non-issue. Plus, sessionStorage is cleared when the tab closes. |
| T-08-09 | Elevation of Privilege | New code path bypassing PocketBase per-user rules? | accept | Phase 8 introduces zero PocketBase queries. The only side effect is sessionStorage I/O. Auth boundary unchanged. |
